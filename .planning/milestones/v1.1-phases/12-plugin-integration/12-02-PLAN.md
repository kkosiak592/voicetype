**1. Create `src/lib/updater.ts`** — a React hook `useUpdater` that manages the full update lifecycle:

```typescript
import { useState, useEffect, useRef, useCallback } from 'react';
import { check, type Update } from '@tauri-apps/plugin-updater';
import { relaunch } from '@tauri-apps/plugin-process';
import { invoke } from '@tauri-apps/api/core';

export type UpdateStatus =
  | 'idle'           // No update activity
  | 'checking'       // Checking for update
  | 'available'      // Update found, waiting for user action
  | 'downloading'    // Download in progress
  | 'ready'          // Downloaded, waiting for restart
  | 'error';         // Check or download failed

export interface UpdateState {
  status: UpdateStatus;
  version: string;
  body: string;              // Release notes markdown
  downloaded: number;        // Bytes downloaded
  contentLength: number;     // Total bytes (0 if unknown)
  errorMessage: string;
}

export interface UseUpdaterReturn {
  state: UpdateState;
  checkNow: () => Promise<void>;
  startDownload: () => Promise<void>;
  cancelDownload: () => void;
  restartNow: () => Promise<void>;
  restartLater: () => void;
  dismiss: () => void;
}
```

Implementation details for `useUpdater()`:
- Store the `Update` object from `check()` in a ref so `startDownload` can call `update.downloadAndInstall()` on it
- `checkNow`: Sets status to 'checking', calls `check()` from `@tauri-apps/plugin-updater`. If update exists, set status='available' with version/body. If none, set status='idle'. On error, set status='error' with message.
- `startDownload`: Calls `update.downloadAndInstall((progress) => {...})` on the stored Update object. The progress callback receives `{ event: 'Started' | 'Progress' | 'Finished', data: { contentLength?, chunkLength? } }`. Track cumulative downloaded bytes by summing chunkLength on each 'Progress' event. Set status='downloading' before starting, status='ready' when the promise resolves.
- `cancelDownload`: Not directly supported by the plugin API — set a cancelled ref flag. When the download promise resolves/rejects after cancel, ignore the result and reset to 'available'.
  NOTE: The actual download continues in the background (plugin doesn't support cancellation). The UI just stops showing progress.
  LIMITATION: The JS plugin `Update` object is stored in a React ref. When the settings window closes and the component unmounts, this ref is destroyed. On reopen, the hook reinitializes to 'idle' and `checkNow()` runs again. If a download was in progress, the user will see the banner reset to 'available' state — they can re-trigger the download. This is a known JS plugin API limitation (no persistent handle to reconnect).
- `restartNow`: Before relaunching, check if the user is actively recording/dictating. The backend emits `pill-state` events with values "recording", "processing", or "idle" to the pill window. Since the settings window cannot listen to pill events directly, use a simple approach: call a new lightweight Rust command `is_pipeline_active` that checks the tray state or the LevelStreamActive flag. If active, show "Waiting for dictation to finish..." and poll every 500ms. Once idle, call `relaunch()` from `@tauri-apps/plugin-process`. If not active, relaunch immediately. This implements the CONTEXT.md decision: "Check if user is actively recording/dictating before relaunching — defer until idle if active."
- `restartLater`: Sets status back to 'ready' but stops showing the banner (set a `dismissed` state). The update is already installed by `downloadAndInstall` — next app restart applies it.
- `dismiss`: Hides the banner for current session. On next settings open / next check cycle, if update is still available, the banner reappears.
- On mount: Call `checkNow()` after a 4-second delay using setTimeout.
- Periodic check: Set up setInterval with 4-hour interval (4 * 60 * 60 * 1000 ms). Clear on unmount. Only fires if status is 'idle' or 'error' (don't interrupt downloading/ready states).
- Also emit a Tauri event to the backend when an update is found, so the tray can be updated:
  `invoke('set_update_available', { available: true })` — this is a new simple command.
