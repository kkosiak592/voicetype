## Summary

This phase polishes the existing pill overlay from a functional MVP into a premium-feeling widget. The work is purely CSS and React — no new Tauri events, no backend changes, no new npm dependencies. The key technical risk is glassmorphism: `backdrop-filter: blur()` does NOT function on transparent Tauri windows on Windows due to a confirmed WebView2 bug (Microsoft issue #4945, open as of August 2025). The workaround is simulated glassmorphism using a semi-transparent dark background with layered inset box-shadows and a visible border, producing the glass aesthetic without relying on blur.

The frequency bar animation must move from per-render random jitter (`Math.random()` inside render) to time-based sinusoidal offsets using `requestAnimationFrame` and `useRef`. Each bar gets a phase offset so bars undulate like a wave. The bar heights are composited from: (a) the sinusoidal idle wave, scaled by (b) the incoming `level` prop. This gives organic motion at rest and reactive motion during speech.

State transitions (scale-in entrance, cross-fade content during state changes, scale-out dismissal) are achievable with CSS `@keyframes` and `transition` on `transform` + `opacity`. The existing `pill.css` already uses `@property` and `@keyframes` — this pattern extends naturally. No animation library (Framer Motion, etc.) is needed; pure CSS handles every required transition.

**Primary recommendation:** Simulate glassmorphism with `rgba` backgrounds + layered box-shadows (no `backdrop-filter`). Drive bar animation with `requestAnimationFrame` sinusoidal interpolation in `FrequencyBars.tsx`. All other transitions via CSS `@keyframes` in `pill.css`.

## Critical Constraint: backdrop-filter Does Not Work

**This is the most important finding in this research.**

`backdrop-filter: blur()` does not work when `transparent: true` is set on a Tauri window on Windows. This is a confirmed WebView2 rendering engine bug (Microsoft internal tracking ID: AB#52983110, GitHub WebView2Feedback issue #4945). As of August 2025 the issue remains open with no fix timeline. Microsoft's official explanation: "The blurring effect uses the surrounding pixels to achieve the effect. Since WebView2 enables a transparent background color, now the blurring effect has no surrounding pixels to use to achieve the blur."

The pill window config already has `"transparent": true`. Changing it is not viable — it would break the invisible-when-hidden state the pill depends on. Therefore `backdrop-filter` is categorically off the table.

**Glassmorphism without backdrop-filter** is well-established and looks credible when done with:
- Semi-transparent dark background: `rgba(15, 10, 30, 0.82)` (deep indigo-dark)
- Top-edge inset highlight: `inset 0 1px 0 rgba(255,255,255,0.12)` (simulates light catching glass)
- Outer diffuse shadow: `0 8px 32px rgba(0,0,0,0.45)` (depth and lift)
- 1px border: `rgba(99,102,241,0.3)` or `rgba(255,255,255,0.15)` (glass edge)

This is the approach used when backdrop-filter is unavailable in contexts like Jetpack Compose (no native blur), Electron apps with layered windows, and CSS fallbacks.

**Note:** `window-vibrancy` (the tauri-apps/window-vibrancy crate) could apply OS-level acrylic blur on Windows 10/11 for true blurred backgrounds. However, it has documented bad drag performance on Windows 10 v1903+ and Windows 11 22000, and introduces a Rust dependency. Since this phase is UI-only polish, and the pill is frequently dragged, this option carries too much risk. Simulated glassmorphism is the correct call.

### Pattern 1: Simulated Glassmorphism (no backdrop-filter)

**What:** A dark semi-transparent pill that reads as "glass" through shadow layering and a fine border.
**When to use:** Any element on a transparent Tauri/WebView2 window on Windows.

```css
/* pill.css — base glass style for all visible states */
.pill-glass {
  background: rgba(15, 10, 30, 0.82);
  border: 1px solid rgba(99, 102, 241, 0.28);
  box-shadow:
    0 8px 32px rgba(0, 0, 0, 0.5),
    inset 0 1px 0 rgba(255, 255, 255, 0.10),
    inset 0 -1px 0 rgba(0, 0, 0, 0.20);
}
```

The inset top highlight simulates light catching the top edge of glass. The deep outer shadow provides lift. The indigo border tints the edge with the design accent color.

### Pattern 2: Entrance/Exit via CSS scale + opacity

**What:** Pill appears by scaling from a near-zero dot to full size. Disappears in reverse. No JS needed.
**When to use:** `pill-show` and `pill-hide` events.

The current approach uses `opacity` only (`transition-opacity duration-300`). Replace with a combined `transform` + `opacity` transition. The key insight: use `transform-origin: center` so the pill scales from its center, not a corner.

```css
/* pill.css */
@keyframes pill-enter {
  from {
    opacity: 0;
    transform: scale(0.15);
  }
  to {
    opacity: 1;
    transform: scale(1);
  }
}

@keyframes pill-exit {
  from {
    opacity: 1;
    transform: scale(1);
  }
  to {
    opacity: 0;
    transform: scale(0.15);
  }
}

.pill-entering {
  animation: pill-enter 220ms cubic-bezier(0.34, 1.56, 0.64, 1) forwards;
}

.pill-exiting {
  animation: pill-exit 180ms ease-in forwards;
}
```

The `cubic-bezier(0.34, 1.56, 0.64, 1)` is a slight overshoot spring — makes the pill "pop" into existence rather than linearly scale. On exit, `ease-in` is faster and snappier (no lingering).

**Pill.tsx integration:** Add an `animState` ref (`"entering" | "visible" | "exiting" | "hidden"`) separate from `displayState`. When `pill-show` fires, set `animState = "entering"`, listen for `animationend`, then set `"visible"`. When `pill-hide` fires, set `animState = "exiting"`, listen for `animationend`, then call `appWindow.hide()` and set `"hidden"`.

### Pattern 3: Sinusoidal FrequencyBars with requestAnimationFrame

**What:** Replace per-render `Math.random()` jitter with a time-based sinusoidal loop. Each bar oscillates at its own frequency and phase so the bars flow like a wave rather than jitter chaotically.
**When to use:** Whenever `displayState === "recording"`.

```tsx
// components/FrequencyBars.tsx
import { useEffect, useRef, useState } from "react";

const BAR_COUNT = 12;
// Each bar gets a unique base frequency and phase offset
const BAR_CONFIG = Array.from({ length: BAR_COUNT }, (_, i) => ({
  freq: 1.2 + (i % 5) * 0.35,      // 1.2 to 3.0 Hz range
  phase: (i / BAR_COUNT) * Math.PI * 2, // spread phases across 2π
  amp: 0.55 + Math.sin((i / BAR_COUNT) * Math.PI) * 0.45, // bell curve: center bars taller
}));

export function FrequencyBars({ level }: { level: number }) {
  const [heights, setHeights] = useState(() => BAR_CONFIG.map(() => 4));
  const rafRef = useRef<number>(0);
  const startTimeRef = useRef<number>(performance.now());

  useEffect(() => {
    function tick(now: number) {
      const t = (now - startTimeRef.current) / 1000; // seconds
      const MIN_H = 4;   // px — always visible
      const MAX_H = 22;  // px — maximum height at level=1

      setHeights(BAR_CONFIG.map((bar) => {
        // sin oscillates -1..1; shift to 0..1 and scale by level
        const wave = (Math.sin(t * bar.freq * Math.PI * 2 + bar.phase) + 1) / 2;
        // Idle: small residual wave even at level=0; reactive at level>0
        const reactive = level * bar.amp * wave;
        const idle = 0.08 * wave;
        return MIN_H + (MAX_H - MIN_H) * Math.min(reactive + idle, 1);
      }));

      rafRef.current = requestAnimationFrame(tick);
    }

    rafRef.current = requestAnimationFrame(tick);
    return () => cancelAnimationFrame(rafRef.current);
  }, [level]);

  return (
    <div className="flex items-end gap-[2px]" style={{ height: "22px" }}>
      {heights.map((h, i) => (
        <div
          key={i}
          className="w-[3px] rounded-full flex-shrink-0"
          style={{
            height: `${h}px`,
            background: `linear-gradient(to top, #6366f1, #c084fc)`,
            transition: "height 40ms ease-out",
          }}
        />
      ))}
    </div>
  );
}
```

Note: `transition: height 40ms ease-out` on each bar provides micro-smoothing between RAF frames without blurring the responsiveness.

```css
/* pill.css */
.pill-checkmark-draw {
```

### Anti-Patterns to Avoid

- **`backdrop-filter: blur()` on any pill element:** Will silently fail on Windows WebView2 transparent windows. No error, no blur — just no effect. Confirmed WebView2 bug.
- **`Math.random()` inside render for bar animation:** Causes bars to jump erratically on every React re-render (when any event fires), not on a smooth time cadence. Must be moved to RAF loop outside React state.
- **Animating `width` or `height` for entrance/exit:** Triggers layout reflow, causes jitter. Use `transform: scale()` only.
- **`window.hide()` before exit animation completes:** Cuts the animation short. The `animationend` event (or a correctly sized timeout) must gate the hide call.
- **Mixing `transition` and `animation` on the same property:** e.g., having both `transition: opacity` and `animation: pill-exit` controlling opacity causes unpredictable results. Pick one per property per element.
- **`requestAnimationFrame` loop running when pill is hidden:** Wastes CPU. The RAF loop in `FrequencyBars` should only run when `displayState === "recording"`. The `useEffect` cleanup cancels it; ensure the effect re-runs when `displayState` changes.

## Common Pitfalls

### Pitfall 1: backdrop-filter Has No Effect (Silent Failure)

**What goes wrong:** Developer applies `backdrop-blur-md` or `backdrop-filter: blur(8px)` to the pill. On Windows with `transparent: true`, the pill renders with no blur — but no console error, no warning. It simply doesn't blur.
**Why it happens:** WebView2 transparent mode removes surrounding pixel context the blur algorithm needs. Microsoft-confirmed bug, unresolved as of August 2025.
**How to avoid:** Do not use `backdrop-filter` on any element inside the pill window. Use the simulated glassmorphism approach (rgba background + layered box-shadows) documented above.
**Warning signs:** Blur missing only on Windows, blur works in browser devtools/hot-reload but not in built app.

### Pitfall 2: Entrance Animation Direction Mismatch

**What goes wrong:** The pill scales from `0` (fully invisible dot), which means the border and glow are also invisible during scale-in. The animation feels like a "pop" rather than growing from a dot.
**Why it happens:** `scale(0)` collapses the element to a point; the glass border becomes too thin to see before ~30% scale. This is actually correct behavior — it just needs the right easing curve to feel intentional.
**How to avoid:** Use `scale(0.1)` to `scale(1)` rather than `0` to `1`, and pair with `cubic-bezier(0.34, 1.56, 0.64, 1)` (spring overshoot). The 0.1 starting point keeps the pill's physical form recognizable during entry.
**Warning signs:** Animation feels "poppy" in a bad way or the pill seems to appear from nowhere rather than grow.

### Pitfall 3: RAF Loop Leaks / Continues When Hidden

**What goes wrong:** `FrequencyBars` mounts when `displayState === "recording"`, but if the parent conditionally renders it, React unmount + remount cycles can cause multiple RAF loops running simultaneously (each triggering state updates).
**Why it happens:** If the effect cleanup doesn't fire reliably (e.g., StrictMode double-invoke in dev), two loops run simultaneously and fight over `setHeights`.
**How to avoid:** Use a single `rafRef = useRef<number>(0)`, always call `cancelAnimationFrame(rafRef.current)` before starting a new RAF in the effect, and in the cleanup function. Verify with React DevTools that the effect runs exactly once in production mode.
**Warning signs:** Bars animate erratically in rapid bursts; CPU usage stays elevated after pill hides.

### Pitfall 4: Content Flash During State Transitions

**What goes wrong:** When switching from "recording" to "processing", the bars disappear instantly and the processing dots appear instantly — there's a visible frame where the pill content is empty.
**Why it happens:** React conditionally renders `{displayState === "recording" && <FrequencyBars />}` — the old content unmounts and new content mounts in the same render cycle. CSS transitions on the container don't apply to children individually.
**How to avoid:** Two approaches: (1) Cross-fade by keeping both elements in DOM and using `opacity` transitions with absolute positioning inside the pill. (2) Brief CSS `transition: opacity 100ms` on the content wrapper, toggled by state. The CONTEXT.md decision is "bars shrink down while pulsing glow fades in" — this implies approach (1) where bars are already mounted with opacity control.
**Warning signs:** Visible blank/white flash when hotkey is released.

### Pitfall 5: Window Size Not Updated for Larger Pill

**What goes wrong:** If pill dimensions increase from 120x40 (e.g., to accommodate the checkmark, wider glass border), the Tauri window size in `tauri.conf.json` still reports 120x40. Content clips or gets cut off.
**Why it happens:** The WebView viewport is capped to the physical window size. Tauri doesn't auto-resize to content.
**How to avoid:** If dimensions change, update both `width`/`height` in `tauri.conf.json` AND the initial position calculation in `Pill.tsx` (which currently uses hardcoded `120` for centering). Keep `resizable: false` — just update the static values.
**Warning signs:** Pill content clipped on right/bottom edge; pill appears misaligned from expected center position.

## Open Questions

1. **Pill window dimensions**
   - What we know: Current window is 120×40 px. The checkmark SVG and glass border may want more room. The `initPosition` function hardcodes `120` for centering.
   - What's unclear: Whether 120×40 is tight for the new content (particularly the scale-up entrance — a 120px wide pill scaling from 10% starts at 12px, which may be too small to read as a "dot").
   - Recommendation: Size to 160×48 px. Update `tauri.conf.json` `width`/`height` AND the `(screenW - 120) / 2` centering calculation in `Pill.tsx`. This is a Claude's Discretion item — recommend 160×48.

2. **Cross-fade timing for recording→processing transition**
   - What we know: CONTEXT.md says "bars shrink down while pulsing glow fades in, no content flash."
   - What's unclear: Whether bars should animate to zero height (all bars shrink) or opacity fade as a group.
   - Recommendation: Fade bars as a group (opacity 1→0, 120ms) while the processing dots fade in (opacity 0→1, 120ms) with 60ms overlap. This is simpler than animating individual bar heights and achieves the same visual goal.

3. **RAF loop and React StrictMode**
   - What we know: StrictMode double-invokes effects in development, which can cause two RAF loops to start.
   - What's unclear: Whether the `cancelAnimationFrame(rafRef.current)` cleanup pattern is sufficient in StrictMode.
   - Recommendation: Pattern is standard and correct — StrictMode cleanup fires before second invocation. No special handling needed. Verify once in dev.
