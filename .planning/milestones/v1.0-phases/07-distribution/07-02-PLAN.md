
**Component structure:**

1. **Header section:** Brief context — one sentence explaining offline transcription needs a model file (per CONTEXT.md locked decision: "Brief context: one sentence explaining that offline transcription needs a model file").

2. **GPU detection badge:** Show detected hardware:
   - GPU detected: green badge "NVIDIA GPU Detected"
   - No GPU: neutral badge "CPU Mode"

3. **Model cards:** Two side-by-side cards (responsive: stack on narrow).
   For each model card:
   - Model name (e.g., "Large v3 Turbo")
   - File size (e.g., "574 MB")
   - Quality descriptor (e.g., "Best accuracy" / "Fastest")
   - Hardware requirement (e.g., "Requires NVIDIA GPU" / "Works on any CPU")
   - "Recommended" badge on the GPU-recommended model (per CONTEXT.md: "recommended one highlighted based on GPU detection")
   - Download button

   Model data hardcoded in FirstRun (not fetched via list_models — first-run flow is self-contained):
   ```typescript
   const MODELS = [
       { id: 'large-v3-turbo', name: 'Large v3 Turbo', size: '574 MB', quality: 'Best accuracy', requirement: 'Requires NVIDIA GPU' },
       { id: 'small-en', name: 'Small (English)', size: '190 MB', quality: 'Fastest', requirement: 'Works on any CPU' },
   ];
   ```

4. **Download state machine:** State: `idle | downloading | validating | complete | error`
   - `idle`: Show download buttons on each card
   - `downloading`: Show progress bar on selected card, disable other card's button
   - `validating`: Show "Verifying checksum..." text briefly (between stream end and Finished event)
   - `complete`: Show success message, enable autostart, then call `onComplete()` after 1 second
   - `error`: Show error message with "Retry" button

5. **Download logic:** Uses `invoke` + `Channel` from `@tauri-apps/api/core`:
   ```typescript
   import { invoke, Channel } from '@tauri-apps/api/core';

   const onEvent = new Channel<DownloadEvent>();
   onEvent.onmessage = (msg) => {
       switch (msg.event) {
           case 'started': setTotalBytes(msg.data.totalBytes); setDownloadState('downloading'); break;
           case 'progress': setDownloadedBytes(msg.data.downloadedBytes); break;
           case 'finished': setDownloadState('complete'); break;
           case 'error': setErrorMsg(msg.data.message); setDownloadState('error'); break;
       }
   };
   await invoke('download_model', { modelId: selectedModel, onEvent });
   ```

6. **Autostart enablement on completion:** When download state transitions to `complete`, call `invoke('enable_autostart')` as fire-and-forget (best effort — catch and log errors but do not block the flow). This implements the locked user decision: "Auto-start with Windows enabled by default". The call should happen BEFORE calling `onComplete()`:
   ```typescript
   // In the complete state handler or useEffect watching downloadState === 'complete':
   try {
       await invoke('enable_autostart');
   } catch (e) {
       console.warn('Failed to enable autostart:', e);
       // Non-blocking — user can toggle autostart later in settings
   }
   // Then after a brief delay, call onComplete()
   setTimeout(() => onComplete(), 1000);
   ```

7. **Progress bar:** Show percentage, downloaded/total MB, and a simple CSS progress bar.
   Percentage: `(downloadedBytes / totalBytes) * 100`. Handle totalBytes=0 gracefully (show indeterminate — just "Downloading..." with bytes counter).

8. **Styling:** Use existing Tailwind classes consistent with settings panel design (dark mode support via `dark:` variants). Center the content in the settings window. Use indigo accent color consistent with existing UI.

9. **Cancel support:** Add a "Cancel" button during download. On cancel, simply abort the invoke by navigating back to idle state. Note: the Rust download_model will continue in background — on next attempt it will overwrite the .tmp file. For v1 this is acceptable. Set a ref to track cancellation and ignore subsequent events after cancel.
