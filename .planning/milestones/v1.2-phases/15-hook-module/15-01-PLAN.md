Create `src-tauri/src/keyboard_hook.rs` with the following structure. This is the hook infrastructure — the hook_proc body will be filled in by Plan 02.

**Public API:**
- `pub enum HookEvent { Pressed, Released }` — events sent from hook callback to dispatcher
- `pub struct HookHandle` — owns the hook thread ID (AtomicU32) and join handle; provides `uninstall()` method
- `pub fn install(app_handle: tauri::AppHandle) -> Result<HookHandle, String>` — spawns hook thread + dispatcher thread, returns handle

**Static state:**
- `static HOOK_TX: OnceLock<std::sync::mpsc::SyncSender<HookEvent>>` — set once during install(), used in hook_proc to send events without allocation

**Hook thread (std::thread::spawn, named "keyboard-hook"):**
1. Call `GetCurrentThreadId()` and store in `Arc<AtomicU32>` shared with HookHandle
2. Call `SetWindowsHookExW(WH_KEYBOARD_LL, Some(hook_proc), None, 0)` — hmod=None and dwThreadId=0 for global scope
3. If SetWindowsHookExW fails, log error and return (thread exits)
4. Run `GetMessageW` loop until it returns 0 (WM_QUIT) or -1 (error)
5. After loop exits, call `UnhookWindowsHookEx(hook)` for cleanup
6. Log "Hook thread exiting" at info level

**hook_proc (unsafe extern "system" fn):**
For this plan, implement ONLY the skeleton:
```rust
unsafe extern "system" fn hook_proc(ncode: i32, wparam: WPARAM, lparam: LPARAM) -> LRESULT {
    if ncode < 0 {
        return CallNextHookEx(None, ncode, wparam, lparam);
    }

    let kb = &*(lparam.0 as *const KBDLLHOOKSTRUCT);

    // Skip injected events (prevents infinite loop when we inject VK_E8 in Plan 02)
    if (kb.flags.0 & LLKHF_INJECTED.0) != 0 {
        return CallNextHookEx(None, ncode, wparam, lparam);
    }

    // TODO(Plan 02): Modifier state machine + debounce + Start menu suppression
    // For now, pass all events through
    CallNextHookEx(None, ncode, wparam, lparam)
}
```

**Dispatcher thread (std::thread::spawn, named "hook-dispatcher"):**
Spawned inside `install()`. Receives `rx: Receiver<HookEvent>` and `app_handle: tauri::AppHandle`. Loops on `rx.recv()` and calls `handle_shortcut()` via the app handle. For now, log received events at info level since handle_shortcut integration comes in Plan 03.

The dispatcher needs to create synthetic `ShortcutEvent` equivalents. Since `handle_shortcut()` takes `&tauri_plugin_global_shortcut::ShortcutEvent`, and we need to construct those from our HookEvent, create a helper:
```rust
fn dispatch_hook_event(app: &tauri::AppHandle, event: HookEvent) {
    // Plan 03 will wire this to handle_shortcut().
    // For now, log the event for verification.
    match event {
        HookEvent::Pressed => log::info!("Hook dispatch: Pressed"),
        HookEvent::Released => log::info!("Hook dispatch: Released"),
    }
}
```

**HookHandle::uninstall():**
1. Load thread_id from AtomicU32
2. If non-zero, call `PostThreadMessageW(thread_id, WM_QUIT, WPARAM(0), LPARAM(0))`
3. Log "Hook uninstall requested" at info level

**Implement Drop for HookHandle** as a safety net — calls `uninstall()` to prevent dangling hooks if HookHandle is dropped without explicit cleanup.

**Channel setup:**
- Use `std::sync::mpsc::sync_channel::<HookEvent>(32)` — bounded channel, 32 capacity
- Store `tx` in `HOOK_TX` OnceLock
- Pass `rx` to dispatcher thread

**Key constraints (from RESEARCH.md):**
- MUST use `std::thread::spawn`, NOT tokio task (WH_KEYBOARD_LL requires stable OS thread)
- hook_proc MUST NOT: allocate, lock Mutex, call async functions, sleep
- hook_proc CAN: read/write AtomicBool/AtomicU32, call try_send on mpsc channel
- hmod parameter to SetWindowsHookExW MUST be None (not HMODULE from GetModuleHandle) when dwThreadId=0
- GetMessageW loop is REQUIRED — without it, the hook is silently removed by Windows

**Required imports:**
