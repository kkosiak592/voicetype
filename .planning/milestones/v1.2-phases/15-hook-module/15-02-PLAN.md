Add a `ModifierState` struct using atomics for thread-safe access from the hook callback. This struct is stored in a `static` (since hook_proc is a static callback) and tracks:

```rust
use std::sync::atomic::{AtomicBool, AtomicU32, Ordering};

/// Modifier state tracked by the hook callback.
/// All fields are atomic — hook_proc is called from a single thread but
/// must be lock-free for timing guarantees (HOOK-02: <5ms).
struct ModifierState {
    ctrl_held: AtomicBool,      // Any Ctrl (left or right) is down
    win_held: AtomicBool,       // Any Win (left or right) is down
    shift_held: AtomicBool,     // Any Shift is down (for exact-match check)
    alt_held: AtomicBool,       // Any Alt is down (for exact-match check)
    combo_active: AtomicBool,   // Ctrl+Win combo is currently active (recording)
    first_key_time: AtomicU32,  // Timestamp (ms) of first modifier keydown in potential combo
}

static STATE: ModifierState = ModifierState {
    ctrl_held: AtomicBool::new(false),
    win_held: AtomicBool::new(false),
    shift_held: AtomicBool::new(false),
    alt_held: AtomicBool::new(false),
    combo_active: AtomicBool::new(false),
    first_key_time: AtomicU32::new(0),
};
```

Replace the TODO in hook_proc with the full state machine. The logic:

**On any KEYDOWN/SYSKEYDOWN:**
1. Read `vk = VIRTUAL_KEY(kb.vkCode as u16)`
2. Check message type: `is_down = wparam.0 as u32 == WM_KEYDOWN || wparam.0 as u32 == WM_SYSKEYDOWN`
3. Check message type: `is_up = wparam.0 as u32 == WM_KEYUP || wparam.0 as u32 == WM_SYSKEYUP`

**Shift/Alt tracking (for exact-match enforcement):**
- VK_LSHIFT, VK_RSHIFT, VK_SHIFT: on down set shift_held=true, on up set shift_held=false
- VK_LMENU, VK_RMENU, VK_MENU: on down set alt_held=true, on up set alt_held=false

**Ctrl keydown (VK_LCONTROL or VK_RCONTROL):**
- Set ctrl_held = true
- If win_held is already true AND combo is not yet active:
  - Check debounce: `(kb.time - first_key_time) <= 50` (50ms window)
  - Check exact match: shift_held == false AND alt_held == false
  - If both pass: set combo_active = true, send `HookEvent::Pressed` via `HOOK_TX.get().unwrap().try_send()`
- If win_held is false: store `first_key_time = kb.time` (Ctrl was first key)
- Always: `CallNextHookEx` (Ctrl keydown is never suppressed)

**Win keydown (VK_LWIN or VK_RWIN):**
- If combo_active is already true: inject VK_E8 mask key, return `LRESULT(1)` to suppress (repeated Win keydown during active combo)
- Set win_held = true
- If ctrl_held is already true AND combo is not yet active:
  - Check debounce: `(kb.time - first_key_time) <= 50`
  - Check exact match: shift_held == false AND alt_held == false
  - If both pass: set combo_active = true, inject VK_E8 mask key, send `HookEvent::Pressed`, return `LRESULT(1)` to suppress Win keydown
- If ctrl_held is false: store `first_key_time = kb.time` (Win was first key)
- If combo did NOT fire: `CallNextHookEx` (Win alone passes through — MOD-05)

**Ctrl keyup (VK_LCONTROL or VK_RCONTROL):**
- Set ctrl_held = false
- If combo_active is true: set combo_active = false, send `HookEvent::Released`
- Always: `CallNextHookEx`

**Win keyup (VK_LWIN or VK_RWIN):**
- Set win_held = false
- If combo_active is true: set combo_active = false, send `HookEvent::Released`, return `LRESULT(1)` to suppress Win keyup (prevents Start menu — MOD-04)
- If combo is NOT active: `CallNextHookEx` (Win alone releases normally — Start menu opens — MOD-05)

**All other keys:**
- `CallNextHookEx` — pass through without modification. Extra keys during active recording are ignored per Claude's discretion (CONTEXT.md).

**CRITICAL: Use Relaxed ordering for all atomic operations.** The hook_proc runs on a single thread so no cross-thread ordering concerns apply within the callback. The only cross-thread communication is via try_send on the channel, which has its own synchronization.

**CRITICAL: try_send, not send.** If the channel is full (32 capacity), drop the event rather than block. Log a warning if try_send fails. This guarantees the hook callback never blocks (HOOK-02).

**CRITICAL: The debounce comparison must handle u32 wraparound.** `GetTickCount` wraps at ~49 days. Use wrapping subtraction: `kb.time.wrapping_sub(first_key_time) <= 50`.

Add the `inject_mask_key()` function that sends a VK_E8 (0xE8) KEYDOWN via SendInput. This function is called from hook_proc when suppressing the Win key during an active Ctrl+Win combo.

```rust
/// Inject VK_E8 (unassigned key) via SendInput to break the Win key's
/// Start menu activation sequence. The system sees an intervening keypress
/// between Win-down and Win-up, so it does not open the Start menu.
///
/// Uses KEYDOWN only (standard AHK MenuMaskKey behavior).
/// If Windows 11 testing shows Start menu still opens, add KEYUP injection
/// as fallback (see 15-RESEARCH.md Open Question 1).
unsafe fn inject_mask_key() {
    let input = INPUT {
        r#type: INPUT_KEYBOARD,
        Anonymous: INPUT_0 {
            ki: KEYBDINPUT {
                wVk: VIRTUAL_KEY(0xE8),  // VK_E8: unassigned, stable across Windows versions
                wScan: 0,
                dwFlags: KEYBD_EVENT_FLAGS(0),  // KEYDOWN (dwFlags=0 means keydown)
                time: 0,
                dwExtraInfo: 0,
            },
        },
    };
    SendInput(&[input], std::mem::size_of::<INPUT>() as i32);
}
```

**Verify inject_mask_key is called from hook_proc in these scenarios:**
1. Win KEYDOWN when ctrl_held and combo just activated → inject before returning LRESULT(1)
2. Win KEYDOWN when combo_active is already true (repeated Win press during recording) → inject before returning LRESULT(1)

**The LLKHF_INJECTED guard at the top of hook_proc (from Plan 01) prevents infinite recursion:** When SendInput injects VK_E8, the hook fires again for that synthetic event, but the LLKHF_INJECTED flag is set, so hook_proc returns CallNextHookEx immediately.

**Do NOT inject VK_E8 for Ctrl keydown.** Ctrl does not trigger the Start menu. Only Win key events need suppression.

```rust
}
```
