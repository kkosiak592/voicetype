## Summary

This phase bundles four UI quality-of-life improvements against an already-working codebase. The work is primarily additive (new History section, new IPC commands) with one significant removal (multi-profile switcher system) and two Rust-side fixes (tray duplicate, double-click handler). All four items are independent and can be planned as separate plans within the phase.

The profile simplification is the most structurally invasive change. The current code has a two-profile system (`structural-engineering` / `general`) with hard-coded profile structs in `profiles.rs`, IPC commands (`get_profiles`, `set_active_profile`), and frontend components (`ProfileSwitcher.tsx`, `ProfilesSection.tsx`). This all gets replaced with a single `initial_prompt` field persisted to `settings.json`. The `Profile` struct, `ActiveProfile` managed state, and `get_profiles`/`set_active_profile` commands need to be simplified or replaced ŌĆö but `all_caps` and `corrections` must survive since they're used in `pipeline.rs`.

Transcription history storage should use a separate `history.json` file in the app data directory (same location as `settings.json`). This avoids bloating `settings.json` with up to 50 text entries and keeps the history concern isolated. The Rust side gets new `save_history_entry` and `get_history` IPC commands; the frontend gets a new `HistorySection` component.

The tray duplicate bug is almost certainly caused by the `relaunch()` path in the updater: Windows holds the old tray icon for a brief period after process exit, and if the new process registers immediately, both icons appear. The fix is to call `app.tray_by_id("tray").map(|t| t.destroy())` (or equivalent cleanup) before relaunch, or add a small delay. The `tauri.conf.json` has no `trayIcon` config section ŌĆö so the config-vs-code duplication theory from the GitHub issue does not apply here.

The double-click handler in Tauri 2 uses `TrayIconBuilder::on_tray_icon_event` with a `TrayIconEvent::DoubleClick` match arm. This is Windows-only per Tauri documentation. The existing `tray.rs` `build_tray()` function needs an `on_tray_icon_event` chain added.

**Primary recommendation:** Implement as four focused plans ŌĆö (1) tray duplicate fix + double-click handler, (2) profile simplification, (3) history storage + IPC, (4) history UI panel.

### Pattern 1: Tauri 2 Tray Double-Click Handler

**What:** Add `on_tray_icon_event` to the `TrayIconBuilder` chain in `build_tray()`
**When to use:** This is the only supported way to handle tray click events in Tauri 2

```rust
// Source: https://v2.tauri.app/learn/system-tray/
use tauri::tray::{TrayIconBuilder, TrayIconEvent};

TrayIconBuilder::with_id("tray")
    .icon(tauri::image::Image::from_bytes(ICON_IDLE)?)
    .tooltip("VoiceType - Idle")
    .menu(&menu)
    .show_menu_on_left_click(false)
    .on_tray_icon_event(|tray, event| {
        if let TrayIconEvent::DoubleClick { button, .. } = event {
            if button == tauri::tray::MouseButton::Left {
                let app = tray.app_handle();
                if let Some(w) = app.get_webview_window("settings") {
                    let _ = w.show();
                    let _ = w.set_focus();
                }
            }
        }
    })
    .on_menu_event(|app, event| { /* existing handler */ })
    .build(app)?;
```

**Note:** `TrayIconEvent::DoubleClick` is Windows-only per Tauri docs. Since this app is Windows-only (NSIS bundle target), no platform guard needed.

### Pattern 2: History File Persistence

**What:** Separate `history.json` in app data dir, mirroring the `settings.json` approach
**When to use:** New file-based persistence

```rust
// Mirrors load_settings_from_disk / flush_settings in lib.rs
fn history_path(app: &tauri::AppHandle) -> std::path::PathBuf {
    app.path().app_data_dir()
        .unwrap_or_default()
        .join("history.json")
}

fn load_history(app: &tauri::AppHandle) -> Vec<HistoryEntry> {
    std::fs::read_to_string(history_path(app))
        .ok()
        .and_then(|s| serde_json::from_str(&s).ok())
        .unwrap_or_default()
}

fn save_history(app: &tauri::AppHandle, entries: &[HistoryEntry]) {
    if let Ok(path) = app.path().app_data_dir().map(|d| d.join("history.json")) {
        let _ = std::fs::write(path, serde_json::to_string_pretty(entries).unwrap_or_default());
    }
}
```

```rust
// NEW command: save the vocabulary prompt
#[tauri::command]
fn set_vocabulary_prompt(app: tauri::AppHandle, prompt: String) -> Result<(), String> {
    // Update in-memory settings and flush
    let mut json = read_settings(&app)?;
    json["vocabulary_prompt"] = serde_json::Value::String(prompt.clone());
    write_settings(&app, &json)?;
    // Update ActiveProfile.initial_prompt for pipeline.rs to read
    let profile_state = app.state::<profiles::ActiveProfile>();
    let mut guard = profile_state.0.lock().map_err(|e| e.to_string())?;
    guard.initial_prompt = prompt;
    Ok(())
}
```

### Pattern 5: Reading initial_prompt After Profile Simplification

`pipeline.rs` currently reads `initial_prompt` from `ActiveProfile` managed state. This pattern is preserved ŌĆö the `ActiveProfile` struct is simplified (removes `id`, `name`, multi-profile infra) but keeps `initial_prompt`, `corrections`, and `all_caps`. The pipeline read path is **unchanged**.

### Anti-Patterns to Avoid

- **Removing `ActiveProfile` managed state entirely:** `pipeline.rs` reads `initial_prompt` and `all_caps` from it at transcription time. The struct must survive, just simplified.
- **Merging history into `settings.json`:** Will cause SettingsState lock contention and bloat ŌĆö use separate file.
- **Reading history from disk on every transcription:** Load once into `HistoryState` managed state on startup; append in memory and flush to disk after each new entry.
- **Removing `save_corrections` IPC:** It is still needed ŌĆö the corrections dictionary persists on a per-profile key today (`corrections.structural-engineering`). After simplification, persist under a fixed key like `corrections.default`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Timestamp formatting | Custom date formatter | `format!` with epoch division (ms -> human) or keep as ms and format in JS | Epoch ms is trivially formatted in JS with `new Date(ms).toLocaleString()` |
| Clipboard write | Custom clipboard impl | `navigator.clipboard.writeText()` in the browser context | Available in Tauri webview; no Rust-side arboard needed for settings UI |
| History persistence locking | Mutex<Vec<HistoryEntry>> with complex sync | Simple: load-append-save pattern under a dedicated `HistoryState(Mutex<Vec<HistoryEntry>>)` | Writes are infrequent (one per dictation); serialized by Mutex is fine |

## Common Pitfalls

### Pitfall 1: Tray Duplicate on Relaunch
**What goes wrong:** Two tray icons appear in the system tray simultaneously after update-triggered relaunch
**Why it happens:** Windows defers tray icon cleanup by a few hundred ms after process exit. The new process calls `build_tray()` before the OS removes the old icon.
**How to avoid:** The actual fix depends on the root cause observed. Two candidates:
  1. The `tauri.conf.json` has no `trayIcon` section (confirmed), so config duplication is ruled out.
  2. Most likely: `relaunch()` from `@tauri-apps/plugin-process` does not give Windows time to clean up. The cleanup should happen on process exit (Tauri destroys managed state), but the tray icon may persist briefly.
**Fix approach:** Destroy the tray icon explicitly before calling `relaunch()`. Add a Rust IPC command `destroy_tray()` that calls `app.tray_by_id("tray").map(|t| t.destroy())`, invoke it from the JS updater before `relaunch()`. Alternatively, investigate whether the icon is registered twice due to a hot-reload dev restart (dev-only issue).
**Warning signs:** Issue reproducible only after update install + restart, not on clean launch.

### Pitfall 2: `set_active_profile` Called After Removal
**What goes wrong:** Frontend code (App.tsx, store.ts) still calls `set_active_profile` or reads `activeProfile` from the store after the backend IPC is removed.
**Why it happens:** Multiple call sites ŌĆö `App.tsx` loadSettings reads `savedActiveProfile`, `ProfilesSection.tsx` invokes `set_active_profile` on mount.
**How to avoid:** Audit all callers before removing the command. Remove `activeProfile` from `DEFAULTS` in `store.ts` and `SectionId` 'profiles' from `Sidebar.tsx`. The `save_corrections` command must survive but its internal key changes from `corrections.{profile_id}` to `corrections.default`.

### Pitfall 3: `corrections` Key Migration in settings.json
**What goes wrong:** Existing users have corrections saved under `corrections.structural-engineering` or `corrections.general`. After simplification, the new single key `corrections.default` reads nothing.
**Why it happens:** The `save_corrections` IPC uses `format!("corrections.{}", profile_id)` as the key.
**How to avoid:** On startup, if `corrections.default` is absent but `corrections.structural-engineering` or `corrections.general` exists, migrate: read the old key, write to `corrections.default`, delete old keys. Or simply treat missing `corrections.default` as empty ŌĆö user will re-enter corrections. Given this is a personal-use app, migration is a nice-to-have, not mandatory.

### Pitfall 4: `all_caps` Lost After Profile Removal
**What goes wrong:** `all_caps` is stored per-profile as `profiles.structural-engineering.all_caps`. After simplification there is no profile ID to scope it.
**Why it happens:** `set_all_caps` uses `format!("profiles.{}.all_caps", profile_id)` as the key.
**How to avoid:** Change storage key to flat `all_caps` in settings.json. Read it at startup via `read_settings`. Keep the `set_all_caps` IPC command (it's still needed).

### Pitfall 5: History Not Saved on First Entry
**What goes wrong:** History IPC commands registered but `HistoryState` not populated at startup; first entry writes `[entry]` correctly but then second entry sees empty list from disk if state wasn't flushed.
**Why it happens:** If the managed state isn't seeded from disk at startup, the in-memory list starts empty and overwrites the file on next write.
**How to avoid:** In `setup()`, load `history.json` into `HistoryState` managed state immediately after loading settings.

### Pitfall 6: `TrayIconEvent::DoubleClick` Button Check
**What goes wrong:** Double-clicking the right mouse button also opens settings.
**Why it happens:** `TrayIconEvent::DoubleClick` fires for left, right, or middle button. Without a button check, right double-click (which should show context menu) also opens settings.
**How to avoid:** Check `button == tauri::tray::MouseButton::Left` inside the DoubleClick handler.

### append_history IPC (pipeline.rs integration point)

```rust
// Called in pipeline.rs after successful transcription (step 6, before reset_to_idle)
fn append_history(app: &tauri::AppHandle, text: &str, engine: &str) {
    let entry = HistoryEntry {
        text: text.to_string(),
        timestamp_ms: std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .unwrap_or_default()
            .as_millis() as u64,
        engine: engine.to_string(),
    };
    if let Some(state) = app.try_state::<HistoryState>() {
        let mut guard = state.0.lock().unwrap_or_else(|e| e.into_inner());
        guard.insert(0, entry);  // newest first
        guard.truncate(50);
        save_history(app, &guard);
    }
}
```

## Open Questions

1. **Tray duplicate root cause**
   - What we know: `tauri.conf.json` has no `trayIcon` section (rules out config-vs-code duplication). The updater calls `relaunch()` from `@tauri-apps/plugin-process`.
   - What's unclear: Whether the stale icon issue is a dev-mode hot-reload artifact or reproducible in release builds on update install.
   - Recommendation: Implement `destroy_tray` IPC and call it before `relaunch()`. If not reproducible in release, the fix is still harmless and correct practice.

2. **`all_caps` migration for existing users**
   - What we know: Currently stored as `profiles.structural-engineering.all_caps` in settings.json.
   - What's unclear: Whether the user has `all_caps` enabled (if yes, it silently resets to false after this phase without migration).
   - Recommendation: Add a one-time migration in `setup()` that reads `profiles.structural-engineering.all_caps` and writes it to the flat `all_caps` key if the latter is absent.

3. **`corrections` migration for existing users**
   - What we know: Corrections saved under `corrections.structural-engineering`.
   - Recommendation: One-time migration in `setup()`: if `corrections.default` absent and `corrections.structural-engineering` present, copy to `corrections.default`.
