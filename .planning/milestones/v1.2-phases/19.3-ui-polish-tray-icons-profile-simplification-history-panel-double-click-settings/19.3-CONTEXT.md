<decisions>
## Implementation Decisions

### Transcription History
- Each entry stores: transcribed text, timestamp, model used (engine name)
- Retain last 50 entries — oldest dropped when limit reached
- User action: click to copy text to clipboard (no delete, no clear all)
- New "History" section in settings sidebar

### Profile Simplification
- Remove the structural_engineering / general profile switcher concept
- Flatten to a single editable vocabulary prompt field
- Keep existing corrections dictionary editor alongside the prompt field

### Claude's Discretion
- History storage mechanism (separate JSON file vs other approach in app data dir)
- Where the vocabulary prompt field and corrections dictionary live in settings UI (replace Profiles section, merge into General, or other)
- How to present history entries (list layout, timestamp formatting, copy interaction)
- Tray icon duplicate fix approach (investigate root cause — likely related to app restarts or icon registration)
- Double-click event handling on tray icon (Tauri 2 tray event API)

</decisions>

## Existing Code Insights

### Reusable Assets
- `tray.rs`: TrayIconBuilder with `with_id("tray")`, 3 icon states (Idle/Recording/Processing), menu event handler for settings/quit
- `ProfileSwitcher.tsx`: Card-based profile selector — will be removed/replaced
- `ProfilesSection.tsx`: Contains profile switcher + ALL CAPS toggle + corrections dictionary editor — needs restructuring
- `DictionaryEditor.tsx`: Standalone corrections dictionary component — reusable as-is
- `Sidebar.tsx`: Settings navigation with 5 sections — needs "History" added, "Profiles" renamed or replaced

### Established Patterns
- Settings stored via `settings.json` through Rust-side `SettingsStore` (single Mutex-backed store)
- IPC commands follow `invoke('command_name', { params })` pattern
- Sidebar sections are statically defined in `ITEMS` array with id/label/icon
- Tray menu is immutable after creation — rebuilt via `tray.set_menu(Some(new_menu))` on changes

### Integration Points
- Pipeline (`pipeline.rs`) reads `initial_prompt` from active profile — needs to read from new single-prompt source
- `profiles.rs` defines `Profile` struct with `initial_prompt` and `corrections` — needs simplification
- Tray icon built in `build_tray()` during app setup — double-click handler goes here
- History entries created in pipeline after successful transcription — new save point needed
