<blockers>

- encode() in streaming_model.rs feeds ALL accumulated feature frames to ONNX encoder in one call
- Encoder ONNX model has internal 4096-frame limit (attention bias tensor at node_add_10)
- Audio >81.9s (4096 frames / 50fps) overflows this limit
- Fix requires looping through features in windows of [left_context + frame_len + lookahead] per encoder call
- First fix attempt was wrong — need to study C++ reference implementation more carefully
</blockers>

<context>
The key insight: the v2 streaming encoder uses sliding-window attention and is designed to process
frame_len=80 new frames per call, with left_context (16*depth) frames of history and total_lookahead=16
frames of future context. The encode() method must loop through accumulated features in these windows.

The streaming_config.json has: frame_len=80, total_lookahead=16, depth=6/10/14.
left_context = 16 * depth (computed in code, not in config).

Reference: .planning/debug/moonshine-v2-streaming-encoder-broadcast-error.md has full evidence chain.
Key file: src-tauri/patches/transcribe-rs/src/engines/moonshine/streaming_model.rs (encode() method ~line 243)
</context>
