# Phase 19.1: Integrate Moonshine Tiny Model into Main App with VAD Chunking and GPU Support - Research

**Researched:** 2026-03-04
**Domain:** Rust/Tauri transcription engine integration — Moonshine ONNX model wiring into production pipeline
**Confidence:** HIGH

## Summary

Phase 19.1 integrates the Moonshine Tiny ONNX transcription engine into the main VoiceType application as a selectable model option, alongside existing Whisper and Parakeet engines. The benchmark work (Quick Tasks 28-35) already validated performance and wired the full transcription-rs patch, meaning the foundational library code exists and compiles. This phase is purely an integration exercise: expose Moonshine Tiny as a new `TranscriptionEngine` variant, register its managed state, wire the download flow, wire VAD-based chunking for clips exceeding the model's 30s context window, and expose GPU execution-provider selection.

The key constraint is that Moonshine Tiny uses a fundamentally different runtime (ONNX via `transcribe-rs`) than Whisper (whisper-rs C bindings) and Parakeet (parakeet-rs ONNX). The `transcribe-rs` crate is already patched locally (`src-tauri/patches/transcribe-rs`) with GPU execution-provider support and is already compiled under the `bench_extra` feature. Promoting it to the main app requires adding a new `moonshine` feature flag, new managed state, a new `TranscriptionEngine::Moonshine` variant, VAD chunking in the pipeline, a download command for the directory-based model (two ONNX files + tokenizer.json), and frontend model card integration.

Performance data from benchmarks (CPU-only on this developer machine): Moonshine Tiny achieves 191ms avg on 5s clips and 2052ms on 60s clips — faster than Parakeet for short clips and dramatically faster than Whisper. WER is competitive with Whisper Small at 0-9% for short clips with some degradation on longer content. The model only processes up to 30 seconds per inference call without VAD chunking, which is why chunking is a hard requirement for this integration.

**Primary recommendation:** Add a `moonshine` Cargo feature that promotes `transcribe-rs` from bench-only to main-app dependency; create a new `TranscriptionEngine::Moonshine` variant with `Arc<Mutex<MoonshineEngine>>` managed state following the exact Parakeet pattern; implement VAD chunking in `pipeline.rs` (reusing `vad_chunk_audio` logic from the benchmark); add a directory-download command for the two-file ONNX model from `onnx-community/moonshine-tiny-ONNX` on HuggingFace; add a model card in `list_models` and `FirstRun.tsx`.

### VAD Chunking for Moonshine (New transcribe_moonshine.rs)

The transcription function for Moonshine must handle the 30s context window limit by splitting audio with VAD:

```rust
// In src-tauri/src/transcribe_moonshine.rs:
pub fn transcribe_with_moonshine(
    engine: &mut MoonshineEngine,
    audio: &[f32],
) -> Result<String, String> {
    const MOONSHINE_MAX_SAMPLES: usize = 30 * 16000; // 30s at 16kHz

    let chunks: Vec<Vec<f32>> = if audio.len() > MOONSHINE_MAX_SAMPLES {
        // VAD chunking — reuse vad_chunk_audio logic
        crate::vad::vad_chunk_for_moonshine(audio)
    } else {
        vec![audio.to_vec()]
    };

    let mut combined_text = String::new();
    for chunk in &chunks {
        match engine.transcribe_samples(chunk.clone(), None) {
            Ok(result) => {
                if !combined_text.is_empty() && !result.text.trim().is_empty() {
                    combined_text.push(' ');
                }
                combined_text.push_str(result.text.trim());
            }
            Err(e) => return Err(format!("Moonshine inference error: {}", e)),
        }
    }

    let text = combined_text.trim().to_string();
    log::info!("Moonshine transcription: '{}' ({} chunks)", &text[..text.len().min(80)], chunks.len());
    Ok(text)
}
```

### GPU Execution Provider Wiring

Moonshine uses the same `ort` execution provider pattern as Parakeet via the patched `transcribe-rs`. The `CachedGpuDetection` managed state already tracks `parakeet_provider` ("cuda", "directml", or "cpu"). Use that same value to configure Moonshine:

```rust
// In load_moonshine() — follows same pattern as load_parakeet():
pub fn load_moonshine(model_dir: &Path, provider: &str) -> Result<MoonshineEngine, String> {
    use transcribe_rs::engines::moonshine::{MoonshineEngine, MoonshineModelParams, ModelVariant};
    use ort::execution_providers::{CUDAExecutionProvider, CPUExecutionProvider};

    let execution_providers = match provider {
        "cuda" => Some(vec![
            CUDAExecutionProvider::default().build(),
            CPUExecutionProvider::default().build(),
        ]),
        "directml" => {
            // DirectML EP — use DirectMLExecutionProvider
            // (same approach as parakeet-rs patch)
            None // For now — directml may need ort::execution_providers::DirectMLExecutionProvider
        }
        _ => None, // CPU default
    };

    let mut engine = MoonshineEngine::new();
    let mut params = MoonshineModelParams::tiny();
    params.execution_providers = execution_providers;

    engine.load_model_with_params(model_dir, params)
        .map_err(|e| format!("Failed to load Moonshine Tiny model: {}", e))?;

    Ok(engine)
}
```

### Anti-Patterns to Avoid

- **Using MoonshineStreamingEngine for the main app:** Benchmark data proves the streaming engine is 2-3x slower for short clips (5s: 597ms vs 191ms). The batch v1 engine (`moonshine-tiny-ONNX` directory) is correct for hold-to-talk.
- **Passing audio > 30s directly to Moonshine without chunking:** The model has a hard 30s context window limit. Audio over 30s produces garbage output. VAD chunking is mandatory.
- **Creating a separate VAD detector per transcription call in the pipeline:** `VoiceActivityDetector` must be created fresh per use (has internal LSTM state). Fine to create per-call in `transcribe_moonshine.rs` since it's called in `spawn_blocking`.
- **Reusing the single-file `download_model` command for Moonshine:** Moonshine is a directory with multiple ONNX files. Use a dedicated `download_moonshine_tiny_model` command following the Parakeet multi-file pattern.
- **Locking the outer `MoonshineStateMutex` across inference:** Follow the Parakeet pattern — clone the inner `Arc`, drop the outer lock, then lock the inner `Mutex` inside `spawn_blocking`.

## Common Pitfalls

### Pitfall 1: Moonshine 30-Second Context Window Violation
**What goes wrong:** Audio longer than 30 seconds passed directly to `engine.transcribe_samples()` produces garbled output or silently truncates — no panic, just wrong text.
**Why it happens:** Moonshine Tiny (non-streaming, v1) is trained on 4-30 second segments. The model's attention mask and positional encodings have a hard maximum.
**How to avoid:** In `transcribe_with_moonshine()`, check `audio.len() > 30 * 16000` before calling inference and split with VAD.
**Warning signs:** High WER on longer recordings, output that stops mid-sentence, or short output for long audio.

### Pitfall 2: Wrong Engine Variant (Streaming vs Batch)
**What goes wrong:** Accidentally using `MoonshineStreamingEngine` instead of `MoonshineEngine` in the main app — 3x latency regression.
**Why it happens:** Both engines exist in `patches/transcribe-rs/src/engines/moonshine/`. The streaming engine was added in Quick 33 for benchmark comparison, not for production.
**How to avoid:** Main app always uses `MoonshineEngine` from `engine.rs` with `moonshine-tiny-ONNX/` directory (encoder_model.onnx + decoder_model_merged.onnx). Streaming uses `moonshine-streaming-tiny/` with .ort files.
**Warning signs:** Model directory path contains "streaming" or files have `.ort` extension instead of `.onnx`.

### Pitfall 3: Feature Flag Conflict Between bench_extra and moonshine
**What goes wrong:** Adding `moonshine = ["dep:transcribe-rs"]` may conflict with the existing `bench_extra = ["dep:transcribe-rs", "dep:ort"]` if both features try to declare the same optional dependency.
**Why it happens:** Cargo merges features — two features can both enable the same optional dep without conflict. The conflict would be if the dependency is declared differently.
**How to avoid:** `transcribe-rs` is already declared as optional in `[dependencies]`. Both `moonshine` and `bench_extra` can list `"dep:transcribe-rs"` in their feature arrays — Cargo handles this correctly.
**Warning signs:** `cargo check --features whisper,parakeet,moonshine` fails with "dependency specified twice" error.

### Pitfall 4: MoonshineEngine Not Sync (same issue as Parakeet)
**What goes wrong:** Attempting to use `Arc<MoonshineEngine>` as Tauri managed state without wrapping in Mutex.
**Why it happens:** `transcribe_samples` takes `&mut self` — the engine is not Sync.
**How to avoid:** Use `Arc<Mutex<MoonshineEngine>>` (inner) wrapped in `Mutex<Option<...>>` (outer) — exactly the `ParakeetStateMutex` pattern.

### Pitfall 5: Download Existence Check Must Use a Sentinel File
**What goes wrong:** Checking if the `moonshine-tiny-ONNX/` directory exists returns true even if download was incomplete.
**Why it happens:** The directory is created at download start, but files may not be complete.
**How to avoid:** Check for `decoder_model_merged.onnx` inside the directory (the largest file, last to download). Match `parakeet_fp32_model_exists()` pattern.

### Pitfall 6: Frontend Engine Routing in handleModelSelect
**What goes wrong:** `ModelSection.tsx:handleModelSelect` hardcodes `isParakeetVariant` check — adding Moonshine requires updating the frontend routing logic.
**Why it happens:** Current code in `ModelSection.tsx:62-76` dispatches `set_engine("parakeet")` or `set_engine("whisper")` based on model ID. Moonshine is a third engine.
**How to avoid:** Add `isMoonshineVariant` check that dispatches `set_engine("moonshine", ...)` for `model.id === "moonshine-tiny"`.

### Pitfall 7: VAD Creates New Instance Per Recording (Correct)
**What goes wrong:** Temptation to cache the `VoiceActivityDetector` between recordings for performance.
**Why it happens:** VAD construction takes a few milliseconds.
**How to avoid:** DO NOT cache. The Silero model has internal LSTM state — reusing across sessions causes stale activations. The existing codebase already documents this (vad.rs line comment: "Create a fresh VoiceActivityDetector per recording session"). Creating fresh in transcribe_moonshine is correct.

## Open Questions

1. **DirectML execution provider for Moonshine**
   - What we know: The patched transcribe-rs has `cuda = ["ort/cuda"]` and `directml = ["ort/directml"]` feature flags. `ort::execution_providers::DirectMLExecutionProvider` should be constructable.
   - What's unclear: Whether the DirectML EP import path in transcribe-rs matches the ort 2.0.0-rc.10 API. The parakeet-rs patch has `ExecutionProvider::DirectML` which maps through its own abstraction.
   - Recommendation: For the first implementation, support CUDA + CPU fallback only. DirectML for Moonshine can be added in a follow-up. Non-NVIDIA users get CPU.

2. **Warm-up inference for Moonshine**
   - What we know: Parakeet has a `warm_up_parakeet()` call after load to initialize CUDA context (important for first-inference latency). The benchmark shows Moonshine first inference is similar to subsequent runs.
   - What's unclear: Whether CUDA initialization latency is significant for the ONNX Moonshine model.
   - Recommendation: Add a warm-up call mirroring the Parakeet pattern (`spawn` background thread, run 0.5s silent audio through the model). Can be skipped if latency testing shows it's not needed.

3. **SHA256 checksums for Moonshine ONNX files**
   - What we know: The download pattern requires SHA256 per file for integrity validation (as in `download_parakeet_fp32_model`). The local copies in AppData are already downloaded and validated by transcribe-rs.
   - What's unclear: The exact SHA256 hashes for `encoder_model.onnx`, `decoder_model_merged.onnx`, and `tokenizer.json` from `onnx-community/moonshine-tiny-ONNX`. These must be computed from the HuggingFace LFS pointers or by downloading and hashing.
   - Recommendation: During implementation, either skip SHA256 validation for directory downloads (use size check instead) or compute hashes from the local copies. Size check is lower security but acceptable for a development-stage app. Follow whichever the parakeet pattern does — parakeet omits per-file SHA256 in its download function.

4. **First-run flow inclusion**
   - What we know: `FirstRun.tsx` has a hardcoded `MODELS` array with cards including `gpuOnly` flags.
   - What's unclear: Whether Moonshine should appear in the first-run flow or only in settings (since it is already downloaded on this machine — new users need the download).
   - Recommendation: Include Moonshine Tiny in the first-run card array. Show to all users (not GPU-only — Moonshine runs on CPU just fine). Mark as "fastest" / "108 MB".
