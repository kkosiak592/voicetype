# Phase 20.1: Implement the VAD Chunking Method - Research

**Researched:** 2026-03-06
**Domain:** VAD-based audio chunking for Whisper and Parakeet transcription engines
**Confidence:** HIGH

## Summary

Phase 20.1 generalizes the existing `vad_chunk_for_moonshine()` algorithm in `vad.rs` to work with all three transcription engines (Whisper, Parakeet, Moonshine). Currently, only Moonshine has VAD chunking for audio >30s. Whisper and Parakeet receive the entire trimmed buffer as a single inference call, which scales poorly for 60s+ recordings due to Whisper's O(n^2) self-attention and Parakeet's increasing TDT decoder work on longer sequences.

The codebase already contains the complete chunking algorithm (lines 166-268 of `vad.rs`). The implementation requires: (1) generalizing the function with a configurable max segment duration, (2) wrapping the Whisper and Parakeet engine dispatch in `pipeline.rs` with a chunk loop, and (3) concatenating per-chunk results. Three prior research artifacts unanimously recommend this approach. No new dependencies are needed.

**Primary recommendation:** Generalize `vad_chunk_for_moonshine()` into `vad_chunk_audio(samples, max_segment_secs)` and add chunk loop dispatch in `pipeline.rs` for Whisper and Parakeet engines. Activate only for audio >30s (480,000 samples).

## Architecture Patterns

### Current Pipeline Flow (Before)
```
audio_buffer
  -> vad_trim_silence()
  -> engine.transcribe(full_buffer)
  -> post-processing
  -> clipboard inject
```

### Target Pipeline Flow (After)
```
audio_buffer
  -> vad_trim_silence()
  -> if len > 30s: vad_chunk_audio(samples, max_secs) -> [chunk1, chunk2, ...]
     else: [full_buffer]
  -> for each chunk: engine.transcribe(chunk)
  -> join results with space
  -> post-processing
  -> clipboard inject
```

**When to use:** Any time audio needs splitting at silence boundaries before inference.

### Pattern 2: Chunk Loop in Pipeline Dispatch
**What:** Wrap the engine transcription call in a chunk loop when audio exceeds threshold.
**When to use:** In `pipeline.rs` for the Whisper and Parakeet engine arms.
**Example:**
```rust
// Source: existing pattern in transcribe_moonshine.rs:115-139
const CHUNK_THRESHOLD_SAMPLES: usize = 30 * 16000; // 30s

let chunks = if samples.len() > CHUNK_THRESHOLD_SAMPLES {
    vad::vad_chunk_audio(&samples, 30)
} else {
    vec![samples.clone()]
};

let mut combined = String::new();
for chunk in &chunks {
    let text = transcribe_engine(engine, chunk)?;
    let trimmed = text.trim();
    if !trimmed.is_empty() {
        if !combined.is_empty() {
            combined.push(' ');
        }
        combined.push_str(trimmed);
    }
}
```

### Pattern 3: Whisper State Reuse Across Chunks
**What:** Create a single `WhisperState` from the `WhisperContext` and reuse it for all chunks within one pipeline invocation, or create fresh state per chunk.
**When to use:** When transcribing multiple Whisper chunks sequentially.
**Example:**
```rust
// Fresh state per chunk (safest — no cross-chunk LSTM bleed)
for chunk in &chunks {
    let mut state = ctx.create_state().map_err(|e| e.to_string())?;
    let mut params = FullParams::new(SamplingStrategy::Greedy { best_of: 1 });
    params.set_no_context(true);  // critical: no prior context carryover
    params.set_single_segment(true);
    state.full(params, chunk).map_err(|e| e.to_string())?;
    // ... extract text from segments
}
```

### Anti-Patterns to Avoid
- **Chunking inside spawn_blocking:** The VAD pass should happen BEFORE the `spawn_blocking` call, not inside it. The VAD uses ONNX Runtime (same as Parakeet) and having the chunk list ready before entering the blocking context avoids nesting ONNX sessions.
- **Chunking audio under 30s:** Short clips should bypass VAD chunking entirely. The VAD pass adds 2-5ms overhead and produces a single chunk anyway. Use a simple `samples.len() > THRESHOLD` gate.
- **Shared mutable state between chunks:** Each chunk transcription must be independent. For Whisper, use `set_no_context(true)`. For Parakeet, `transcribe_samples` already takes an owned `Vec<f32>` with no state carryover.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Speech/silence detection | Custom energy-based VAD | `voice_activity_detector` (Silero V5) | Neural VAD handles music, typing, breathing; energy thresholds don't |
| Audio splitting at silence boundaries | Custom sliding window | `vad_chunk_for_moonshine()` generalized | Algorithm already handles edge cases: min segment, max segment, fallback |
| Chunk result concatenation | Custom punctuation-aware merging | Simple space-join | Both engines produce complete sentences per chunk; VAD splits at pauses where spaces naturally occur |

**Key insight:** The chunking algorithm is already battle-tested in the Moonshine path. The only work is making it generic and wiring it into two more engine dispatch paths.

## Common Pitfalls

### Pitfall 1: VAD Session Inside spawn_blocking Conflicts with Parakeet ONNX
**What goes wrong:** If the VAD chunking (which uses ONNX Runtime via `voice_activity_detector`) runs inside the same `spawn_blocking` that holds the Parakeet `Mutex<ParakeetTDT>`, there could be ONNX session conflicts or deadlocks.
**Why it happens:** Both VAD and Parakeet use ort ONNX Runtime. Running them in the same blocking context is safe (they're separate sessions), but running VAD while holding the Parakeet mutex is wasteful.
**How to avoid:** Run `vad_chunk_audio()` BEFORE entering `spawn_blocking`, while only `samples` (not the engine) is needed.
**Warning signs:** Long pauses between recording stop and first chunk transcription.

### Pitfall 2: Double VAD Pass
**What goes wrong:** `vad_trim_silence()` already runs on the audio in `pipeline.rs:155` before engine dispatch. Running `vad_chunk_audio()` creates a second fresh `VoiceActivityDetector` and re-classifies all chunks.
**Why it happens:** Each VAD function creates its own detector to avoid stale LSTM state.
**How to avoid:** Accept the double pass. The cost is ~2-5ms per VAD pass on typical audio. Combining them would couple trim and chunk logic unnecessarily. The current `vad_trim_silence()` removes silence; then `vad_chunk_audio()` finds split points in the remaining audio. They serve different purposes.
**Warning signs:** None significant — this is a conscious design tradeoff.

### Pitfall 3: Whisper single_segment(true) with Chunked Audio
**What goes wrong:** The current Whisper params set `single_segment(true)`, which forces all audio into one segment. For VAD-chunked audio, each chunk IS a single segment (one utterance), so this is actually correct.
**Why it happens:** Chunks are split at silence boundaries and are typically 5-30s of continuous speech.
**How to avoid:** Keep `single_segment(true)` for chunked audio. If chunks could contain multiple utterances (they shouldn't with VAD splitting), you'd need to set it to false.
**Warning signs:** Truncated transcription within a chunk.

### Pitfall 4: Result Ordering and Spacing
**What goes wrong:** Chunks are processed sequentially but their text results need proper concatenation. Double spaces, missing spaces, or leading/trailing whitespace at chunk boundaries.
**Why it happens:** Each engine may return text with different whitespace conventions.
**How to avoid:** Trim each chunk result, skip empty results, join with a single space. This is the exact pattern used in `transcribe_moonshine.rs:125-139`.
**Warning signs:** Double spaces in transcribed text, words running together at chunk boundaries.

### Pitfall 5: Max Segment Duration for Parakeet
**What goes wrong:** Setting max_segment_secs too low for Parakeet wastes inference overhead (each call has fixed startup cost). Setting it too high defeats the purpose of chunking.
**Why it happens:** Parakeet TDT can handle longer audio than Whisper (up to 4-5 minutes), but still benefits from chunking.
**How to avoid:** Use 30s for both Whisper and Parakeet. Whisper's native window is 30s, and Parakeet benchmarks show diminishing returns beyond 30s per segment. This also simplifies the code (one threshold for all engines).
**Warning signs:** No improvement in Parakeet latency for 60s recordings.

### Generalized VAD Chunking Function
```rust
// Source: generalization of vad.rs:166-268
pub fn vad_chunk_audio(samples: &[f32], max_segment_secs: u32) -> Vec<Vec<f32>> {
    const CHUNK_SIZE: usize = 512;
    const SPEECH_THRESHOLD: f32 = 0.5;
    const SILENCE_SPLIT_CHUNKS: usize = 10; // ~320ms of silence triggers split
    let max_segment_samples: usize = max_segment_secs as usize * 16000;
    const MIN_SEGMENT_SAMPLES: usize = 8000; // 0.5 seconds

    let mut vad = match VoiceActivityDetector::builder()
        .sample_rate(16000u32)
        .chunk_size(CHUNK_SIZE)
        .build()
    {
        Ok(v) => v,
        Err(e) => {
            log::error!("VAD chunk: init failed: {} — returning single segment", e);
            return vec![samples.to_vec()];
        }
    };

    // ... identical algorithm to vad_chunk_for_moonshine with max_segment_samples parameterized
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Process full audio buffer in single inference | VAD-chunk at silence boundaries before inference | 2024-2025 (WhisperX, faster-whisper, whisper.cpp) | 30-50% latency reduction for 60s+, eliminates hallucination on long silence regions |
| Custom energy-based silence detection | Neural VAD (Silero V5/V6) | 2023 (Silero V5) | More accurate boundary detection, handles music/noise correctly |
| Fixed-length 30s sliding window (whisper.cpp default) | VAD-aware variable-length chunks | 2024 (whisper.cpp PR #3065) | No mid-word splits, cleaner text boundaries |

**Deprecated/outdated:**
- Fixed 30s sliding window without VAD: whisper.cpp still supports this but recommends VAD mode for long audio
- Overlap-based chunk merging: Unnecessary when splitting at silence boundaries (VAD ensures clean splits)

## Open Questions

1. **Optimal max segment duration for Parakeet**
   - What we know: Parakeet TDT can handle up to 4-5 minutes of audio, but attention cost increases. Whisper's native window is 30s.
   - What's unclear: Whether 30s or 45s produces better Parakeet results. Research suggests 30s is fine.
   - Recommendation: Start with 30s for both engines. Tune later if benchmarks show Parakeet performs better with longer chunks.

2. **Whether to refactor vad_chunk_for_moonshine or create a new function**
   - What we know: The existing function has hardcoded 30s max and is only called from `transcribe_moonshine.rs`.
   - What's unclear: Whether to keep the old function as a thin wrapper or replace all callers.
   - Recommendation: Create `vad_chunk_audio(samples, max_segment_secs)` as the generic function. Replace `vad_chunk_for_moonshine`'s body with a call to `vad_chunk_audio(samples, 30)` to maintain backward compatibility. Then update `transcribe_moonshine.rs` to call the generic function directly.
