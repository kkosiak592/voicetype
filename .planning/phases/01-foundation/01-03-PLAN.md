  <action>
    1. Create `src/lib/store.ts` — a thin wrapper around tauri-plugin-store:
       ```typescript
       import { Store } from '@tauri-apps/plugin-store';

       export interface AppSettings {
         hotkey: string;
         theme: 'light' | 'dark';
         autostart: boolean;
       }

       export const DEFAULTS: AppSettings = {
         hotkey: 'ctrl+shift+space',
         theme: 'light',
         autostart: false,
       };

       let _store: Store | null = null;

       export async function getStore(): Promise<Store> {
         if (!_store) {
           _store = await Store.load('settings.json', {
             defaults: DEFAULTS as unknown as Record<string, unknown>,
             autoSave: 100,
           });
         }
         return _store;
       }
       ```

    2. Create `src/components/HotkeyCapture.tsx`:
       - Display current hotkey in a styled box (e.g., "Ctrl+Shift+Space")
       - On click, enter "listening" state (visual change — border color, "Press a key combo..." placeholder)
       - On keydown (while listening): capture modifier+key combo, normalize to lowercase `+`-separated format matching tauri-plugin-global-shortcut expectations (e.g., "ctrl+shift+space")
       - Exclude modifier-only keystrokes (pressing just Ctrl should not register)
       - On valid capture: call `invoke('rebind_hotkey', { old: currentHotkey, new: capturedCombo })`, then write to store via `store.set('hotkey', capturedCombo)`
       - Exit listening state after successful capture
       - On Escape while listening: cancel without changing
       - Key normalization: use `e.code` for the base key (strip "Key" prefix for letters, "Digit" for numbers), lowercase everything. `e.code === 'Space'` becomes `'space'`, `e.code === 'KeyA'` becomes `'a'`.

    3. Create `src/components/ThemeToggle.tsx`:
       - Read current theme from store on mount
       - Toggle between light and dark
       - Apply theme via CSS class on `document.documentElement`: add/remove `.dark` class
       - Write preference to store immediately on toggle
       - Do NOT use `prefers-color-scheme` media query (broken in Tauri WebView per research — issue #5802)

    4. Create `src/components/AutostartToggle.tsx`:
       - Use `@tauri-apps/plugin-autostart` JS API: `isEnabled()`, `enable()`, `disable()`
       - Simple toggle switch that reflects and controls the autostart state
       - Write preference to store for UI state consistency

    5. Update `src/App.tsx` to build the settings page:
       - Layout: single page (appropriate for Phase 1's 3 settings — per Claude's discretion)
       - Fixed-size window (480x400 per tauri.conf.json — no need for responsive breakpoints)
       - Sections: "Hotkey" with HotkeyCapture, "Appearance" with ThemeToggle, "Startup" with AutostartToggle
       - Use Tailwind for styling. Light mode default. Dark mode via `.dark` class on html element.
       - On mount: load store, apply saved theme, display saved hotkey

    6. Update `src/styles.css`:
       - Tailwind base import
       - Define dark mode variants using the `dark:` prefix with `darkMode: 'class'` strategy (Tailwind v3) or equivalent v4 approach
       - Minimal custom styles — Tailwind utilities for everything

    7. Update `src-tauri/src/lib.rs` — restore saved hotkey on startup:
       - In `setup()`, BEFORE registering the default hotkey, read `settings.json` from the store plugin's data directory to check for a saved hotkey
       - If a saved hotkey exists and differs from the default, register that instead of "ctrl+shift+space"
       - Use `app.path().app_data_dir()` to locate the store file, or use the store plugin's Rust API if available
       - If reading fails (first launch, corrupted file), fall back to registering "ctrl+shift+space"
       - This ensures the user's custom hotkey is restored on restart without the frontend needing to re-register it

    NOTE on hotkey restore: The simplest approach may be to read the JSON file directly with serde_json from the store's known path (`{app_data_dir}/settings.json`), since the Rust-side store API may require async setup. Check if `tauri_plugin_store` exposes a synchronous Rust read API usable inside `setup()`. If not, read the file directly — this is a one-time startup read, not ongoing persistence.
