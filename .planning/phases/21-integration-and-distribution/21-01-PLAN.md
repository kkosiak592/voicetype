<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Manual integration tests — modifier desync recovery and Start menu suppression</name>
  <what-built>
    Phases 15-17 implemented the WH_KEYBOARD_LL keyboard hook with modifier state machine, debounce, VK_E8 Start menu suppression, and frontend capture UI. These runtime behaviors cannot be verified by automated tests — they require physical key holds across OS focus changes.
  </what-built>
  <how-to-verify>
    **Prerequisites:**
    - VoiceType is running with Ctrl+Win as the active hotkey (check Settings or settings.json)
    - If using a standard hotkey (e.g., Ctrl+Shift+Space), switch to Ctrl+Win first and restart the app

    **Test A — Modifier State Desync Recovery (Success Criterion 1):**
    1. Start VoiceType with Ctrl+Win hotkey active
    2. Hold Ctrl (do NOT press Win)
    3. Press Alt+Tab to switch to another window (release Alt+Tab but keep holding Ctrl)
    4. Release Ctrl while focus is on the other window
    5. Alt+Tab back to VoiceType
    6. Do NOT press any hotkey — just observe
    7. **Expected:** No recording starts, pill overlay does not appear, tray icon remains idle
    8. **If recording starts:** FAIL — report the exact sequence and what appeared

    **Test B — Start Menu Suppression on Windows 10 (Success Criterion 3):**
    1. Terminate and restart VoiceType (clean hook state)
    2. Press and release Win key alone → **Expected:** Start menu opens (MOD-05)
    3. Close Start menu
    4. Press and release Ctrl+Win together → **Expected:** Dictation activates, Start menu does NOT open (MOD-04)
    5. Stop dictation (release keys)
    6. Press Ctrl first, then press Win → **Expected:** Same as step 4 — dictation activates, no Start menu
    7. Release both keys
    8. Press and release Win key alone again → **Expected:** Start menu opens normally

    **Test C — Start Menu Suppression on Windows 11 (Success Criterion 3, if Win11 machine available):**
    1. Repeat Test B on a Windows 11 machine or VM
    2. Record the Windows 11 build number: run `winver` and note the version (e.g., 23H2, 24H2)
    3. **If Start menu opens during Ctrl+Win:** This is the known VK_E8 KEYDOWN-only issue — report it. A KEYUP injection fix exists in the research document and will need to be applied before distribution.

    **Between each test:** Terminate and restart VoiceType to ensure clean hook state. Do not reuse the same running instance across tests.
  </how-to-verify>
  <resume-signal>
    Report results for each test (A, B, C):
    - "A: PASS" or "A: FAIL — [description]"
    - "B: PASS" or "B: FAIL — [which step failed and what happened]"
    - "C: PASS", "C: FAIL — [details]", or "C: SKIP — no Win11 machine available"

    If Test C fails on Win11 with Start menu opening during Ctrl+Win, report it — a code fix (VK_E8 KEYUP injection) must be applied before proceeding to Task 2. This would require a gap closure plan.
  </resume-signal>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Release build, VirusTotal scan, and distribution gate</name>
  <what-built>
    All v1.2 features are implemented and integration-tested. The release needs to be built via the CI pipeline and the resulting NSIS installer scanned on VirusTotal to confirm no new antivirus detections relative to the v1.1 baseline.
  </what-built>
  <how-to-verify>
    **Step 1 — Establish v1.1 baseline (if not already documented):**
    1. Go to https://github.com/kkosiak592/voicetype/releases/tag/v1.1.0
    2. Download the NSIS installer (VoiceType_1.1.0_x64-setup.exe or similar)
    3. Upload to https://www.virustotal.com
    4. Record: total engines checked, number of detections, which specific engines flagged it
    5. Known Tauri false positives: BitDefender, SecLookup, MaxSecure with "Trojan.Heur.LShot.1" — these are endemic to NSIS installer pattern, not app-specific

    **Step 2 — Build v1.2 release:**
    1. Invoke the `bundle-release` skill (say "bundle release" or "/bundle-release")
    2. Follow the skill's gated workflow: version bump, version file updates, CHANGELOG, commit, tag, push
    3. Wait for the GitHub Actions CI pipeline to complete (~20-40 minutes)
    4. Monitor at: https://github.com/kkosiak592/voicetype/actions
    5. Download the v1.2 NSIS installer from https://github.com/kkosiak592/voicetype/releases

    **Step 3 — Scan v1.2 on VirusTotal:**
    1. Upload the v1.2 NSIS installer to https://www.virustotal.com
    2. Record: total engines checked, number of detections, which specific engines flagged it

    **Step 4 — Compare and gate distribution:**
    - If v1.2 has the SAME engines flagging as v1.1 (e.g., BitDefender "Trojan.Heur.LShot.1"): known false positive, NOT a v1.2 regression → distribution is safe
    - If v1.2 shows NEW engines flagging that did NOT flag v1.1: BLOCKING ISSUE — do NOT distribute
      - Identify which new engines flagged v1.2
      - Check if detection names relate to keylogger/hook patterns ("Heuristic.KEYLOGGER", "Trojan.Hook", "SpyWare.*")
      - Major/trusted AV vendors must be clear: Symantec, ESET, Kaspersky, Malwarebytes, Windows Defender

    **Important:**
    - Scan the CI-built NSIS installer from GitHub Releases, NOT a local build
    - The TAURI_SIGNING_PRIVATE_KEY in release.yml is Ed25519 updater signing for tauri-plugin-updater, NOT Windows Authenticode code signing — the binary is unsigned from AV perspective
    - If new detections appear, submit as false positive to the specific vendor(s), not to VirusTotal (VirusTotal is an aggregator)
  </how-to-verify>
  <resume-signal>

    - "v1.1 baseline: X/Y engines detected [list engines]"

    Then one of:
    - "DIST-01: PASS — no new detections, safe to distribute"
    - "DIST-01: FAIL — new detections found, blocking distribution"
  </resume-signal>
