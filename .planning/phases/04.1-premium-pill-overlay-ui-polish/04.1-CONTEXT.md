# Phase 4.1: Premium Pill Overlay UI Polish - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Visual and interaction polish for the existing pill overlay. Transform the MVP-quality pill (flat black background, basic color swaps, random-jitter bars) into a premium glassmorphism overlay with fluid animations and polished state transitions. No new features or capabilities — purely upgrading the look and feel of the existing pill window states.

</domain>

<decisions>
## Implementation Decisions

### Visual design language
- Glassmorphism: frosted glass effect with backdrop blur, semi-transparent background
- Indigo/purple accent color throughout (consistent with existing processing gradient palette)
- Subtle semi-transparent border (1px, white or indigo tint) for definition against any background
- Processing state: pulsing indigo glow instead of the current spinning conic-gradient border

### Frequency bar animation
- Smooth wave bars with sinusoidal interpolation instead of random jitter per render
- Gradient per bar: indigo at base, lighter purple at tip
- Remove the red recording dot — animated bars alone signal recording state
- Bar count, timing, and overall feel at Claude's discretion

### State transitions
- Show/entrance: scale up from a small dot to full pill size (grows into existence)
- Recording → processing: smooth morph — bars shrink down while pulsing glow fades in, no content flash
- Processing indicator: animated spinner/dots icon instead of "Processing" text
- Success: brief animated checkmark icon, then pill scales down and dismisses
- Error/no-speech: silent dismiss — pill fades away without drawing attention to failed attempts
- Hide/exit: reverse of entrance (scale down to dot, fade out)

### Claude's Discretion
- FFT bands from backend vs simulated RMS for bar animation (pick what gives best visual result for the effort)
- Fixed vs adaptive pill width across states
- Overall pill dimensions (current 120x40 may need to increase for glass effect and animations)
- Exact animation durations and easing curves
- Spinner/dots style for processing state
- Backdrop blur radius and glass opacity values

</decisions>

<specifics>
## Specific Ideas

- Reference apps: BridgeVoice and Wispr Flow for overall premium feel
- The pill should feel like a native OS widget, not a web overlay
- Bars should flow organically — current random jitter looks mechanical
- Success checkmark should be brief (flash, not lingering)
- Error state intentionally silent — don't punish the user for accidental activations

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Pill.tsx`: Main pill component with state machine (hidden/recording/processing/success/error), event listeners for backend communication, drag-to-reposition with position persistence
- `FrequencyBars.tsx`: 15-bar visualizer with fixed BAND_MULTIPLIERS array, takes `level` prop (0-1 normalized RMS)
- `pill.css`: @property-based conic-gradient animation for processing, keyframe animations for success/error flash
- `pill-main.tsx`: Standalone React entry point for pill window

### Established Patterns
- Tailwind CSS for styling with custom CSS for animations that need @property or @keyframes
- Tauri event system for backend → pill communication (pill-show, pill-hide, pill-state, pill-level, pill-result)
- Window management via `@tauri-apps/api/webviewWindow` (show/hide/setPosition/setFocusable)
- Position persistence via `@tauri-apps/plugin-store`

### Integration Points
- Backend emits pill events via Tauri event system — no changes needed to event contract
- pill.html is a separate Vite entry point (multi-page app via rollupOptions)
- Tauri window config in tauri.conf.json (transparent:true, decorations:false, alwaysOnTop:true)
- WebView2 (Chromium) renders the pill — supports backdrop-filter, @property, CSS animations

</code_context>
