  <action>
Rewrite `.github/workflows/release.yml` to support dual CPU/GPU builds. The workflow has two jobs: `build` (matrix) and `cleanup` (post-build).

**Job 1: `build` (matrix strategy)**

```yaml
jobs:
  build:
    runs-on: windows-latest
    permissions:
      contents: write
    strategy:
      fail-fast: false
      matrix:
        variant: [cpu, gpu]
```

Steps (in order):

1. **Checkout** — `actions/checkout@v4` (unchanged)

2. **Setup Node.js** — `actions/setup-node@v4` with `node-version: lts/*` (unchanged)

3. **Install Rust stable toolchain** — `dtolnay/rust-toolchain@stable` (unchanged)

4. **Cache Rust build artifacts** — `Swatinem/rust-cache@v2`:
   - Add `key: ${{ matrix.variant }}` to separate CPU/GPU caches. This prevents cross-contamination (Pitfall 2 from research).
   ```yaml
   with:
     workspaces: src-tauri
     cache-on-failure: true
     key: ${{ matrix.variant }}
   ```

5. **Install CUDA Toolkit (GPU only)** — `Jimver/cuda-toolkit@v0.2.21`:
   - Add `if: matrix.variant == 'gpu'` condition
   - Settings unchanged (cuda: '12.6.3', method: network, same sub-packages)

6. **Install LLVM/clang** — unchanged (both variants need it for whisper-rs bindgen)

7. **Set CUDA architecture targets (GPU only)** — existing step with `if: matrix.variant == 'gpu'` added

8. **Stage CUDA DLLs for bundling (GPU only)** — NEW step:
   ```yaml
   - name: Stage CUDA DLLs for bundling
     if: matrix.variant == 'gpu'
     shell: bash
     run: |
       mkdir -p src-tauri/cuda-libs
       # List available CUDA DLLs for debugging (first CI run)
       ls "$CUDA_PATH/bin/" | grep -iE "cudart|cublas" || true
       cp "$CUDA_PATH/bin/cudart64_12.dll"   src-tauri/cuda-libs/
       cp "$CUDA_PATH/bin/cublas64_12.dll"   src-tauri/cuda-libs/
       cp "$CUDA_PATH/bin/cublasLt64_12.dll" src-tauri/cuda-libs/
       echo "Staged CUDA DLLs:"
       ls -la src-tauri/cuda-libs/
   ```

9. **Install frontend dependencies** — `npm ci` (unchanged)

10. **Build and publish (CPU)** — tauri-action with CPU-specific config:
    ```yaml
    - name: Build and publish (CPU)
      if: matrix.variant == 'cpu'
      uses: tauri-apps/tauri-action@v0
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
        TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
      with:
        tagName: ${{ github.ref_name }}
        releaseName: 'VoiceType ${{ github.ref_name }}'
        releaseBody: |
          ## Downloads

          | Variant | Requirement | Installer |
          |---------|-------------|-----------|
          | **CPU** | Works on any machine | `VoiceType-*-x64-cpu.nsis.exe` |
          | **GPU** | Requires NVIDIA GPU + drivers | `VoiceType-*-x64-gpu.nsis.exe` |
        releaseDraft: false
        prerelease: false
        includeUpdaterJson: true
        updaterJsonPreferNsis: true
        args: --config src-tauri/tauri.cpu.conf.json
    ```

11. **Build and publish (GPU)** — tauri-action with GPU-specific config:
    ```yaml
    - name: Build and publish (GPU)
      if: matrix.variant == 'gpu'
      uses: tauri-apps/tauri-action@v0
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
        TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
      with:
        tagName: ${{ github.ref_name }}
        releaseName: 'VoiceType ${{ github.ref_name }}'
        releaseBody: |
          ## Downloads

          | Variant | Requirement | Installer |
          |---------|-------------|-----------|
          | **CPU** | Works on any machine | `VoiceType-*-x64-cpu.nsis.exe` |
          | **GPU** | Requires NVIDIA GPU + drivers | `VoiceType-*-x64-gpu.nsis.exe` |
        releaseDraft: false
        prerelease: false
        includeUpdaterJson: true
        updaterJsonPreferNsis: true
        args: --features cuda --config src-tauri/tauri.gpu.conf.json
    ```

12. **Rename updater JSON to variant-specific name** — NEW step:
    ```yaml
    - name: Rename updater JSON to variant-specific name
      shell: bash
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      run: |
        TAG="${{ github.ref_name }}"
        VARIANT="${{ matrix.variant }}"
        # Wait briefly for GitHub to process the upload
        sleep 5
        # Download the latest.json that tauri-action uploaded
        gh release download "$TAG" --pattern "latest.json" --output "latest-${VARIANT}.json" --repo "${{ github.repository }}"
        # Upload as variant-specific name (--clobber if other job already uploaded)
        gh release upload "$TAG" "latest-${VARIANT}.json" --clobber --repo "${{ github.repository }}"
        echo "Uploaded latest-${VARIANT}.json"
    ```

**Job 2: `cleanup` (runs after both matrix jobs)**

```yaml
  cleanup:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - name: Remove generic latest.json from release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          TAG="${{ github.ref_name }}"
          # Delete the generic latest.json that both tauri-action jobs uploaded
          gh release delete-asset "$TAG" "latest.json" --yes --repo "${{ github.repository }}" || echo "latest.json already removed"
          # Verify variant-specific files exist
          echo "Release assets:"
          gh release view "$TAG" --repo "${{ github.repository }}" --json assets --jq '.assets[].name'
    ```

**Important implementation notes:**
- `fail-fast: false` is critical — if the CPU build fails, the GPU build should still complete (and vice versa). Users who need only one variant should get it.
- Both jobs write `releaseBody` — the second job to finish will set the final body. This is acceptable since both write identical content. The table format (from the user's discretion) presents both variants equally with clear requirements.
- The CUDA install step is removed from the CPU job path (was previously always-on). The CPU job skips CUDA entirely, producing a faster build.
- LLVM/clang is kept for BOTH variants because whisper-rs uses bindgen for C++ FFI even without CUDA.
- The `releaseName` stays as-is (not variant-suffixed) because there is one release per tag containing both installers.

**Installer naming:**
The user decision requires architecture-first naming: `VoiceType-X.Y.Z-x64-cpu.nsis.exe`. However, tauri-action controls the installer filename via the productName and platform target. The default output is `VoiceType_X.Y.Z_x64-setup.nsis.exe`. To achieve the required naming, we do NOT need to rename the installer — we accept Tauri's default naming convention. The CPU and GPU installers will have different names because they are uploaded as separate assets. If tauri-action produces identical names for both variants, add a post-build rename step similar to the updater JSON rename. Verify on first CI run and adjust if needed.

**NOTE:** If tauri-action produces identical installer filenames for both variants (both named `VoiceType_X.Y.Z_x64-setup.nsis.exe`), the second upload will overwrite the first. In that case, a rename step is needed. The research does not confirm whether `--config` overlay affects the output filename. This is an open question to verify on the first CI test run. If collision occurs, add a rename step in the next gap-closure plan.
