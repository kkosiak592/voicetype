# Phase 19.2: Revamp Model Selection with Benchmark Stats, Recommend Parakeet, Remove distil-large-v3.5 - Research

**Researched:** 2026-03-04
**Domain:** UI/UX model selection revamp + Rust backend model registry cleanup
**Confidence:** HIGH (all findings based on direct codebase inspection; benchmark data is authoritative from src-tauri/benchmark-results.md)

## Summary

Phase 19.2 is a focused cleanup and UX improvement on top of the model work shipped in Phases 19 and 19.1. The benchmark data now exists (Quick tasks 27-37 produced `src-tauri/benchmark-results.md`) and clearly shows parakeet-tdt-v2 is the best balanced model for dictation. The phase has three concrete tasks: (1) remove distil-large-v3.5 from the model registry and all frontend surfaces, (2) surface benchmark stats in model cards so users can make informed choices, and (3) make parakeet the universal recommended model (replacing the current GPU-conditional logic).

The scope is entirely frontend-heavy (ModelSelector.tsx, ModelSection.tsx, FirstRun.tsx) with one small Rust backend change (remove distil-large-v3.5 from `list_models()`, `model_info()`, `check_first_run()`). There are no new Cargo dependencies, no new IPC commands, no new download infrastructure. The `ModelInfo` struct already carries `recommended: bool` and `description: String` — benchmark data can be embedded directly into description strings without any schema changes.

**Primary recommendation:** Remove distil-large-v3.5 from backend and both frontend entry points (ModelSelector/ModelSection and FirstRun), embed benchmark stats in model description strings, and set `recommended: true` for parakeet-tdt-v2-fp32 unconditionally (drop the gpu_mode conditional).

### Interpretation for Model Recommendation

**Distil-large-v3.5** is clearly the worst performer: 15,617 ms average latency with 5.9% WER. It is slower than whisper-large-v3-turbo at similar accuracy, and costs 513 MB. There is no meaningful use case where a user would prefer it over any other listed model. Removal is justified by data.

**Parakeet-tdt-v2** has the best accuracy (5.3% WER) among models that don't require CUDA-only hardware, and it supports DirectML (any discrete GPU), CUDA, and CPU. At 4,494 ms average latency it is respectable. The 2.56 GB download is the cost. As the "best balanced" model it is the correct universal recommendation.

**Moonshine-tiny** is faster overall (2,332 ms) but at 7.2% WER — notably higher error rate at 30s+ clips. It is the best choice for users who prioritize speed and have mostly short (<15s) recordings.

**SenseVoice-small** is not in the current app and is not being added in this phase (no integration code exists).

## Architecture Patterns

### Pattern 1: Model Removal Pattern

To remove distil-large-v3.5 cleanly:

**Backend (3 touch points):**
1. `download.rs` `model_info()`: delete the `"distil-large-v3.5"` match arm
2. `lib.rs` `list_models()`: remove the `models.push(ModelInfo { id: "distil-large-v3.5" ... })` block
3. `lib.rs` `check_first_run()`: remove `distil_v35_exists` variable and its check in `needs_setup`

**Frontend (1 touch point):**
4. `FirstRun.tsx`: remove the `distil-large-v3.5` entry from the `MODELS` array

No IPC schema changes needed. The `download_model` command with `model_id = "distil-large-v3.5"` will simply return an error after removal (no handler needed — the model card won't exist).

### Pattern 2: Benchmark Stats in Description Strings

The `ModelInfo.description` field is a plain `String` in Rust returned to the frontend. It renders as `<p className="mt-0.5 text-xs text-gray-500">{model.description}</p>` in `ModelSelector.tsx`.

Two approaches:
- **Option A (simpler):** Embed stats in description string: `"Best accuracy · 5.3% WER · 4.5s avg · 2.56 GB"`
- **Option B (richer):** Extend `ModelInfo` with structured stat fields

**Recommendation: Option A.** No schema change, no IPC change, immediate. The description is already small text in gray — stats fit naturally. The planner can decide if structured fields are wanted, but the description approach works within the existing API contract.

### Pattern 3: Universal Parakeet Recommendation

Current `list_models()`:
```rust
recommended: gpu_mode,  // only recommends parakeet when NVIDIA GPU present
```

Target:
```rust
recommended: true,  // always recommend parakeet as the best balanced choice
```

Current `check_first_run()` recommended_model:
```rust
recommended_model: if gpu_mode {
    "parakeet-tdt-v2-fp32".to_string()
} else if detection.0.has_discrete_gpu {
    "parakeet-tdt-v2-fp32".to_string()
} else {
    "small-en".to_string()
},
```

Target: unconditionally `"parakeet-tdt-v2-fp32"` for all users (parakeet runs on CPU too, just slower — it is still the best accuracy choice even on CPU).

Also in `list_models()`, `small-en` currently has `recommended: !gpu_mode`. After this change, either make it `false` unconditionally (parakeet is the single recommended model) or keep the logic but the "Recommended" badge should only appear on one model to avoid confusion. Single recommendation is cleaner.

### Pattern 4: FirstRun MODELS Array Sync

The `FirstRun.tsx` `MODELS` array is disconnected from the backend `list_models()` response — it is hardcoded locally. This is an existing divergence that the previous phases accepted. Phase 19.2 needs to:
- Remove `distil-large-v3.5` from `MODELS`
- Update descriptions with benchmark stats (consistent with ModelSection)
- Potentially update `recommendedModel` badge behavior — currently checks `model.id === recommendedModel` from backend, which will return `"parakeet-tdt-v2-fp32"` universally after the backend change. No frontend logic change needed for this part.

## Common Pitfalls

### Pitfall 1: Forgetting check_first_run's distil check
**What goes wrong:** Remove distil from `list_models()` and `model_info()` but forget `check_first_run()`. Existing users with `ggml-distil-large-v3.5.bin` on disk would be shown the first-run setup screen again.
**How to avoid:** Search for all occurrences of `distil` in `lib.rs` and `download.rs` before committing.
**Warning signs:** `needs_setup` returns `true` when it should be `false` for users who had distil installed.

### Pitfall 2: Description truncation in ModelSelector
**What goes wrong:** Long description strings overflow the single-line `<p>` in the model card, getting cut off or wrapping awkwardly.
**How to avoid:** Keep description strings under 60 chars or verify the UI renders multi-line gracefully. The current style uses `text-xs` (12px) in a card that is `px-4 py-3` — there is room for 2 lines.

### Pitfall 3: recommended badge on multiple models
**What goes wrong:** If both `parakeet-tdt-v2-fp32` and `moonshine-tiny` have `recommended: true`, both show the "Recommended" badge, confusing the user.
**How to avoid:** Set `recommended: true` only for parakeet. Set all other models to `recommended: false`.

### Pitfall 4: FirstRun recommended_model defaulting to small-en for CPU users
**What goes wrong:** CPU users see "small-en" recommended in FirstRun even after the backend change, because the small-en entry still has `recommended: !gpu_mode` in `list_models()`.
**How to avoid:** Change `check_first_run()` to always return `"parakeet-tdt-v2-fp32"` as `recommended_model`. Change `list_models()` `small-en` entry to `recommended: false`.

### Pitfall 5: Users with distil model installed
**What goes wrong:** After removing distil from the list, users who previously downloaded it have an orphaned `ggml-distil-large-v3.5.bin` file and their settings.json may still say `whisper_model_id: "distil-large-v3.5"`. On next startup, `set_model` or startup model loading would fail.
**How to avoid:** The startup model loading already has fallback logic — if the saved model_id is not recognized, it falls back gracefully. Verify this by checking the startup model resolution path in `lib.rs`. If `whisper_model_id = "distil-large-v3.5"` is saved and the model is removed from `list_models()`, the user sees the model as gone, and the UI should recover to another model. Test this edge case explicitly.

## Open Questions

1. **Should description format change in FirstRun.tsx too?**
   - What we know: FirstRun has its own `quality` and `size` fields in the local MODELS array (not from backend)
   - What's unclear: Whether FirstRun should show benchmark WER stats too, or keep its current simpler card format
   - Recommendation: Keep FirstRun cards simpler (size + one-line quality description) since FirstRun is a first-impression flow — overwhelming new users with WER% numbers may create decision paralysis. Focus stat richness on the ModelSection settings view.

2. **What happens to users who already have distil-large-v3.5 downloaded and selected?**
   - What we know: `whisper_model_id` in settings.json may be `"distil-large-v3.5"`. The startup model loading path in `lib.rs` reads this. After removal, `model_info("distil-large-v3.5")` returns `None`.
   - What's unclear: Whether the startup path gracefully falls back or panics/errors. Need to check the `set_model` and startup loading flow around line ~1395 in lib.rs.
   - Recommendation: Add explicit check — if saved model_id is `"distil-large-v3.5"`, migrate settings to `"small-en"` or `"large-v3-turbo"` depending on which file exists. Or accept that the user will need to re-select a model (non-fatal — ModelSection handles no-model state).

3. **Should the "Recommended" badge appear in FirstRun cards?**
   - What we know: FirstRun already renders the recommended badge based on `model.id === recommendedModel` from backend. After the backend change, this will highlight parakeet universally.
   - What's unclear: Whether a CPU-only user should see parakeet as "Recommended" even though the 2.56 GB download is slow on CPU.
   - Recommendation: Yes — parakeet has the best accuracy and runs on CPU (just slower). The download cost is a one-time event. Keep universal recommendation.
