## Summary

This phase replaces the 24-bar DOM-based `FrequencyBars.tsx` with a `WaveformCanvas.tsx` component using Canvas2D. The frontend work is straightforward: create a canvas element, run a RAF loop drawing 3 bezier curves per frame with `shadowBlur` bloom passes using `globalCompositeOperation: "lighter"`. The canvas is only 170x38px — the absolute pixel cost of shadowBlur at this size is negligible even though the MDN optimization guide warns against it for general use. The warning applies to large canvases with many elements; at this tiny resolution, 3 bloom passes are well under 1ms per frame.

The backend work adds 16-bin FFT magnitude extraction to `pill.rs`. The key discovery: `rustfft` 6.4.1 is already in the dependency graph as a transitive dependency of `rubato` (via `realfft`). Similarly, `realfft` itself is already present. This means we can use `rustfft` directly in `Cargo.toml` without downloading any new crates — it will resolve from the already-compiled dependency. The FFT runs on the last 256 samples from the audio buffer (16ms at 16kHz), applies a Hann window to reduce spectral leakage, bins the first 128 output magnitudes into 16 groups, normalizes to 0-1 range, and emits alongside or replacing the existing RMS scalar.

The primary risk is the event protocol change: `pill-level` currently emits a raw `f32` scalar. Changing it to emit a struct with `{ rms: f32, bins: Vec<f32> }` requires updating `Pill.tsx` event listener typing and the `WaveformCanvas` component interface. This is low risk but must be coordinated — both Rust and React sides must change together.

**Primary recommendation:** Build `WaveformCanvas.tsx` as a Canvas2D component using the existing RAF + levelRef pattern from `FrequencyBars.tsx`, add FFT emission to `pill.rs` using `rustfft` (already available as transitive dep), and update the event protocol in one coordinated change.

### Pattern 1: Canvas RAF Loop with levelRef (already established in codebase)

**What:** Mount canvas, build RAF loop that reads data from refs (never from props directly), draw per frame, cancel on unmount.

**When to use:** Anytime you need per-frame canvas drawing without React re-renders on every frame.

**Example (based on FrequencyBars.tsx existing pattern):**
```typescript
// Source: FrequencyBars.tsx existing code + Canvas2D API
export function WaveformCanvas({ level, bins }: WaveformCanvasProps) {
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const rafRef = useRef<number>(0);
  const levelRef = useRef<number>(level);
  const binsRef = useRef<number[]>(bins);

  useEffect(() => { levelRef.current = level; }, [level]);
  useEffect(() => { binsRef.current = bins; }, [bins]);

  useEffect(() => {
    const canvas = canvasRef.current;
    if (!canvas) return;

    // HiDPI scaling — MUST happen once on mount
    const dpr = window.devicePixelRatio || 1;
    const W = canvas.offsetWidth;
    const H = canvas.offsetHeight;
    canvas.width = W * dpr;
    canvas.height = H * dpr;
    const ctx = canvas.getContext("2d")!;
    ctx.scale(dpr, dpr);

    const startTime = performance.now();

    function tick(now: number) {
      const t = (now - startTime) / 1000;
      const lv = levelRef.current;
      const bins = binsRef.current;
      ctx.clearRect(0, 0, W, H);
      drawWaveform(ctx, W, H, t, lv, bins);
      rafRef.current = requestAnimationFrame(tick);
    }

    rafRef.current = requestAnimationFrame(tick);
    return () => cancelAnimationFrame(rafRef.current);
  }, []);

  return <canvas ref={canvasRef} style={{ width: "160px", height: "30px" }} />;
}
```

### Pattern 2: Canvas2D Bloom via Multiple Draw Passes

**What:** Draw the waveform curve multiple times at different `shadowBlur` radii and `globalAlpha` values with `globalCompositeOperation: "lighter"` for additive blending.

**When to use:** Creating neon/bloom glow effects on small canvases where shadowBlur cost is acceptable.

**Example:**
```typescript
// Source: Canvas2D API — MDN + prior research artifact
function drawWaveform(ctx: CanvasRenderingContext2D, W: number, H: number,
                       t: number, lv: number, bins: number[]) {
  const amplified = Math.pow(lv, 0.55); // same non-linear mapping as FrequencyBars

  // Build control points from FFT bins (16 bins → 3 curves with bell envelope)
  const points = computeControlPoints(W, H, t, amplified, bins);

  // Save default composite
  ctx.globalCompositeOperation = "source-over";

  // Core line (thin, bright)
  ctx.globalAlpha = 0.9;
  ctx.shadowBlur = 0;
  drawCurves(ctx, points, W, H, 1.5);

  // Bloom pass 1 — tight glow
  ctx.globalCompositeOperation = "lighter";
  ctx.globalAlpha = 0.6;
  ctx.shadowBlur = 8;
  ctx.shadowColor = "#06b6d4";
  drawCurves(ctx, points, W, H, 2);

  // Bloom pass 2 — medium glow
  ctx.globalAlpha = 0.3;
  ctx.shadowBlur = 16;
  drawCurves(ctx, points, W, H, 3);

  // Bloom pass 3 — wide diffuse halo
  ctx.globalAlpha = 0.15;
  ctx.shadowBlur = 24;
  drawCurves(ctx, points, W, H, 5);

  // Reset state
  ctx.globalCompositeOperation = "source-over";
  ctx.globalAlpha = 1;
  ctx.shadowBlur = 0;
}
```

### Pattern 4: FFT Computation in pill.rs

**What:** On each ~30fps tick, take the last 256 samples from the audio buffer, apply a Hann window, run rustfft, take the first 128 magnitude values, bin them into 16 buckets, normalize, emit.

**Example:**
```rust
// Source: rustfft 6.4.1 docs + Hann window formula
use rustfft::{FftPlanner, num_complex::Complex};

fn compute_fft_bins(buf: &[f32], n_bins: usize) -> Vec<f32> {
    let fft_size = 256usize;
    if buf.len() < fft_size {
        return vec![0.0; n_bins];
    }

    // Take last fft_size samples
    let window = &buf[buf.len() - fft_size..];

    // Apply Hann window to reduce spectral leakage
    let mut complex_buf: Vec<Complex<f32>> = window
        .iter()
        .enumerate()
        .map(|(i, &s)| {
            let hann = 0.5 - 0.5 * (2.0 * std::f32::consts::PI * i as f32 / (fft_size - 1) as f32).cos();
            Complex { re: s * hann, im: 0.0 }
        })
        .collect();

    // Run FFT (FftPlanner should be reused; store in state or use lazy_static)
    let mut planner = FftPlanner::<f32>::new();
    let fft = planner.plan_fft_forward(fft_size);
    fft.process(&mut complex_buf);

    // First half is the meaningful spectrum (second half is mirror)
    let half = fft_size / 2; // 128
    let magnitudes: Vec<f32> = complex_buf[..half]
        .iter()
        .map(|c| (c.re * c.re + c.im * c.im).sqrt())
        .collect();

    // Bin into n_bins groups (logarithmic or linear grouping)
    let bin_size = half / n_bins; // 128 / 16 = 8
    let mut bins = vec![0.0f32; n_bins];
    for (b, chunk) in magnitudes.chunks(bin_size).take(n_bins).enumerate() {
        bins[b] = chunk.iter().cloned().fold(0.0f32, f32::max);
    }

    // Normalize: find peak and scale to 0-1
    let peak = bins.iter().cloned().fold(0.0f32, f32::max).max(1e-6);
    bins.iter_mut().for_each(|v| *v = (*v / peak).min(1.0));
    bins
}
```

**Important:** `FftPlanner` should NOT be created on every tick (wastes allocation). Store it as part of the level stream state or use `once_cell`/`OnceLock` for a static planner instance.

### Pattern 5: Tauri Event Protocol Change

**What:** Change `pill-level` from emitting raw `f32` to emitting a struct with both RMS and FFT bins.

**Rust side:**
```rust
#[derive(Clone, serde::Serialize)]
struct PillLevel {
    rms: f32,
    bins: Vec<f32>,
}

// In start_level_stream():
let payload = PillLevel { rms: level, bins: compute_fft_bins(&buf, 16) };
let _ = app.emit_to("pill", "pill-level", payload);
```

**Frontend side (Pill.tsx):**
```typescript
type PillLevelPayload = { rms: number; bins: number[] };
appWindow.listen<PillLevelPayload>("pill-level", (e) => {
  setLevel(e.payload.rms);
  setBins(e.payload.bins);
}).then((u) => unlisteners.push(u));
```

### Anti-Patterns to Avoid

- **Creating FftPlanner on every tick:** Allocates and recomputes plan 30x per second. Cache it in the `start_level_stream` closure or as a shared struct.
- **Using `{ alpha: false }` on canvas:** The pill window is transparent — the canvas must keep alpha enabled (`getContext("2d")` defaults to alpha: true). Using alpha: false would fill the canvas with black, hiding the pill glass background.
- **Drawing on un-scaled canvas:** Missing `devicePixelRatio` scaling produces blurry lines on HiDPI displays. The 1px core waveform line will look 2px thick and fuzzy without DPI scaling.
- **Using `ctx.filter = "blur()"` instead of shadowBlur:** MDN marks `ctx.filter` as "Limited Availability" — not baseline. Stick with `shadowBlur` which is universally supported.
- **Restarting RAF loop on level/bins prop change:** Use refs (levelRef, binsRef) to feed animation data, exactly like FrequencyBars.tsx does. Props trigger re-renders; refs do not.
- **Emitting raw `Vec<f32>` as the entire event payload:** Tauri serializes the payload as the complete JSON value. A raw Vec<f32> works, but wrapping in a struct future-proofs the protocol and allows adding `rms` alongside `bins` without breaking the API.

## Common Pitfalls

### Pitfall 1: FftPlanner Created per Tick

**What goes wrong:** A new `FftPlanner` is created every 33ms inside the level stream loop. Each call to `plan_fft_forward(256)` allocates internal buffers and computes twiddle factors. At 30fps, this is 30 allocations/sec and GC pressure on the Rust allocator.

**Why it happens:** The documentation shows `FftPlanner::new()` as the entry point without emphasizing it should be cached.

**How to avoid:** Create `FftPlanner` once when `start_level_stream` begins. Pass it (or the planned `Arc<dyn Fft<f32>>`) into the async loop closure.

**Warning signs:** High CPU usage during recording; visible jitter in level emission.

### Pitfall 2: DevicePixelRatio Not Applied

**What goes wrong:** The canvas renders correctly on 1x displays but shows blurry waveforms on HiDPI monitors (any modern laptop with 1.5x or 2x scaling).

**Why it happens:** CSS sizes the canvas at 160x30px but the canvas bitmap is also 160x30 pixels. On a 2x DPI display, 160 logical pixels = 320 physical pixels, so the browser stretches the canvas bitmap, making lines blurry.

**How to avoid:**
```typescript
const dpr = window.devicePixelRatio || 1;
canvas.width = logicalWidth * dpr;
canvas.height = logicalHeight * dpr;
ctx.scale(dpr, dpr);
canvas.style.width = `${logicalWidth}px`;
canvas.style.height = `${logicalHeight}px`;
```

**Warning signs:** Lines look thick/fuzzy on laptop displays; sharp on external monitors at 100% scaling.

### Pitfall 3: shadowBlur State Not Reset After Bloom Passes

**What goes wrong:** After bloom passes set `shadowBlur = 24`, subsequent non-bloom draws (or the core line on the next frame) still have shadowBlur active. This causes every draw call to apply the expensive blur, compounding performance cost and visual artifacts.

**Why it happens:** Canvas context is stateful — `shadowBlur` persists until explicitly reset.

**How to avoid:** At the end of each frame's draw function, reset: `ctx.shadowBlur = 0; ctx.globalAlpha = 1; ctx.globalCompositeOperation = "source-over";`

**Warning signs:** Waveform glows even when level is 0; glow radius seems to change unpredictably between frames.

### Pitfall 4: alpha: false on Transparent Canvas

**What goes wrong:** `canvas.getContext("2d", { alpha: false })` fills the canvas with an opaque black background, defeating the pill's transparent overlay design.

**Why it happens:** MDN optimization guide recommends `alpha: false` for performance when transparency isn't needed. But the pill window IS transparent — the canvas sits on a transparent WebView2 window.

**How to avoid:** Use default `canvas.getContext("2d")` (alpha: true). Verify the pill glass background shows through.

**Warning signs:** Dark/black rectangle appears inside the pill during recording state instead of showing the glass background.

### Pitfall 5: Event Protocol Change Breaks Existing Listener

**What goes wrong:** `pill-level` currently emits a raw `f32`. If the Rust side is changed to emit `{ rms: f32, bins: Vec<f32> }` before the TypeScript side is updated, the `setLevel(e.payload)` call in `Pill.tsx` will receive an object instead of a number, breaking the existing bars.

**Why it happens:** Tauri events are JSON-serialized — changing the struct changes the wire format.

**How to avoid:** Change both Rust (pill.rs) and TypeScript (Pill.tsx + component interface) in the same commit. The component swap from `FrequencyBars` to `WaveformCanvas` can happen in the same change.

**Warning signs:** Level always shows 0 or NaN; TypeScript type errors in the listener callback.

### Pitfall 6: FFT Bins From Near-Silence Producing Visual Noise

**What goes wrong:** When the mic is open but no speech is happening, low-level noise produces non-zero FFT bins. The waveform shows random flickering instead of a calm idle state.

**Why it happens:** RMS normalization: `peak = max(bins).max(1e-6)` divides all bins by the peak of a very small signal, normalizing near-silence to full range.

**How to avoid:** Gate the bins on the RMS level. If `rms < threshold` (e.g., 0.02), emit all-zero bins regardless of FFT output. This mirrors the idle behavior in FrequencyBars.tsx where `levelRef` is used as a gate.

**Warning signs:** Waveform is visually active even when the room is silent.

```rust
// Source: DSP standard formula — no crate needed
fn apply_hann(samples: &[f32]) -> Vec<f32> {
    let n = samples.len();
    samples.iter().enumerate().map(|(i, &s)| {
        let w = 0.5 - 0.5 * (2.0 * std::f32::consts::PI * i as f32 / (n - 1) as f32).cos();
        s * w
    }).collect()
}
```

```typescript
function bellCurve(i: number, count: number): number {
```

### Bezier Curve Through Control Points

```typescript
// Source: Canvas2D API — smooth curve through N points using quadraticCurveTo
function drawSmoothCurve(ctx: CanvasRenderingContext2D,
                          points: {x: number, y: number}[]) {
  if (points.length < 2) return;
  ctx.beginPath();
  ctx.moveTo(points[0].x, points[0].y);
  for (let i = 1; i < points.length - 1; i++) {
    const mx = (points[i].x + points[i + 1].x) / 2;
    const my = (points[i].y + points[i + 1].y) / 2;
    ctx.quadraticCurveTo(points[i].x, points[i].y, mx, my);
  }
  ctx.lineTo(points[points.length - 1].x, points[points.length - 1].y);
  ctx.stroke();
}
```

## Open Questions

1. **Waveform symmetry (Claude's discretion)**
   - What we know: CONTEXT.md leaves this to discretion. Single-sided draws the curve from flat baseline up; mirrored reflects top curve below center.
   - What's unclear: Which looks more JARVIS-like at 170x38px? Mirrored is visually fuller but uses only ±15px vertical space; single-sided uses the full 30px height.
   - Recommendation: Mirrored. At this pill height, single-sided waveform will look thin. Mirrored creates the visual density expected for the JARVIS aesthetic. Implementation: draw the curve, then draw it reflected across center Y with slightly lower opacity.

2. **RMS alongside FFT vs FFT-only (Claude's discretion)**
   - What we know: CONTEXT.md leaves this to discretion. RMS is already computed in `compute_rms()`.
   - What's unclear: Does the frontend need RMS at all if FFT bins are available?
   - Recommendation: Keep both — emit `{ rms: f32, bins: Vec<f32> }`. The RMS value is used to gate idle animation and set overall amplitude envelope. FFT bins shape the waveform. They serve different purposes.

3. **FFT update rate (Claude's discretion)**
   - What we know: Current `pill-level` runs at ~30fps (33ms sleep). 60fps would double CPU cost in the Rust async loop.
   - Recommendation: Keep 30fps for the FFT emission. FFT at 30fps on a 256-sample window (16ms of audio) provides adequate temporal resolution for speech formants. The Canvas RAF loop always runs at 60fps regardless — it interpolates between received data points using `t` elapsed time.

4. **Idle animation style (Claude's discretion)**
   - What we know: CONTEXT.md says "very dim glow at idle" but leaves style (breathing pulse, gentle wave, flat line) to discretion.
   - Recommendation: Gentle sine wave at very low amplitude (5% of max height) with a single bloom pass. This signals the component is active without distracting. Flat line risks looking like the waveform is broken.

5. **Recording border treatment (Claude's discretion)**
   - What we know: The current `pill-rainbow-border` uses a full rainbow conic gradient. The new waveform is cyan-to-purple.
   - Recommendation: Switch to cyan-purple border to match. Update `.pill-rainbow-border` to use `#06b6d4, #8b5cf6, #06b6d4` in the conic gradient. The rainbow border looks incongruent with a cyan-purple waveform — two competing color languages in the same pill.
