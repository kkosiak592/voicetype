<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Run manual integration tests for v1.2 hook behavior</name>
  <what-built>
    Phases 15-17 implemented: WH_KEYBOARD_LL hook on a dedicated thread, modifier state machine with debounce, VK_E8 Start menu suppression, rebind/coexistence, and frontend capture UI. All automated verification passed. These three tests cover runtime behaviors unverifiable by unit tests.
  </what-built>
  <how-to-verify>
    **IMPORTANT:** Between each test, fully terminate and restart VoiceType to ensure clean hook state.

    **Pre-check:** Open VoiceType settings and confirm the active hotkey is Ctrl+Win (modifier-only). If it shows a standard hotkey, change to Ctrl+Win and restart.

    ---

    **Test A — Modifier state desync recovery (Success Criterion 1):**

    1. Start VoiceType with Ctrl+Win hotkey active
    2. Hold Ctrl (do NOT press Win)
    3. While holding Ctrl, press Alt+Tab to switch to another window
    4. Release Alt and Tab (keep holding Ctrl)
    5. Release Ctrl (while focus is on the OTHER window, not VoiceType)
    6. Alt+Tab back to VoiceType
    7. Do NOT press any hotkey — just observe

    Expected: No recording starts. Pill overlay does NOT appear. Tray icon remains idle. The hook received the Ctrl-up globally and cleared ctrl_held — no phantom session.

    If recording starts on return (phantom session): FAIL — report this immediately, do not continue.

    ---

    **Test B — Start menu suppression on Windows 10 (Success Criterion 3, OS 1 of 2):**

    1. Press and release Win key alone
       Expected: Start menu opens. Close it.
    2. Press Ctrl+Win together (hold both)
       Expected: Dictation activates (pill overlay appears, recording starts). Start menu does NOT open.
    3. Release both keys
       Expected: Recording stops, transcription runs
    4. Press Win first, then Ctrl within normal typing speed
       Expected: Same behavior as step 2 (debounce handles order)
    5. Press and release Win key alone again
       Expected: Start menu opens normally (MOD-05 preserved)

    If Start menu opens during step 2 or 4: FAIL — report the specific step.

    ---

    **Test C — Start menu suppression on Windows 11 (Success Criterion 3, OS 2 of 2):**

    Repeat all steps from Test B on a Windows 11 machine (or VM).

    **CRITICAL Win11 risk:** The VK_E8 KEYDOWN-only injection may not suppress the Start menu on Win11. If step 2 or 4 fails on Win11 (Start menu opens when it should not), this is the known risk from STATE.md. Report the specific Windows 11 build number (run `winver`). A code fix exists (add KEYUP to inject_mask_key in keyboard_hook.rs) — report the failure so it can be applied before retesting.

    If a second Windows 11 machine or VM is NOT available: report "Win11 untested" — this is a known constraint, not a blocker for distribution on Win10.
  </how-to-verify>
  <resume-signal>Report results for each test (A, B, C) as PASS or FAIL with details. If Win11 is unavailable, state "Win11 untested."</resume-signal>
